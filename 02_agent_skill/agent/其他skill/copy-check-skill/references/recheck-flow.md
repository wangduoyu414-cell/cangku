# Recheck Flow

## 目的

本文件定义修改稿进入复检模式后的比较流程。

复检不是把新稿重新点评一遍，而是要回答四个问题：

1. 上一轮的关键问题是否被解决
2. 是否只是表面更顺，而不是底层更准
3. 是否引入了新的结构性副作用
4. 下一轮最该继续改什么

本文件是复检任务的唯一输出契约。进入复检后，不再套用 `output-contract.md` 的首轮输出格式。

## 复检触发条件

出现以下任一情况，进入复检模式：

- 用户提交同一条文案的修改稿
- 用户明确要求比较前后两个版本
- 用户要求判断“这样改后问题有没有解决”

如果用户更换了分析对象，就不进入复检，而是重新开始一次普通诊断。

## 复检所需输入

复检默认需要以下信息：

- 上一版文案
- 当前版文案
- 上一轮诊断的主问题层级
- 上一轮列出的关键缺陷
- 本轮是否修改了目标或语境

如果缺少上一轮诊断记录，但有前后两个版本，可以先重建上一轮问题，再进入复检。

## 可比性检查

进入复检前先判断两个版本是否还能直接比较。

以下情况视为“可直接比较”：

- 主目标未变
- 分析对象未变，仍是同一条文案或同一片段
- 只是改结构、补证据、调节奏、降结论强度等

以下情况视为“不可直接比较”：

- 从开头改成整篇
- 从引流目标改成成交目标
- 从口播稿改成图文长文，承载任务已变化
- 文案主题已经换掉

不可直接比较时，必须立即退出复检，不再进入后续复检流程。

退出动作固定如下：

1. 先输出“不可直接复检”的一句话说明。
2. 再说明导致不可比的变化点，例如对象变了、主目标变了、承载任务变了。
3. 然后切回首轮路径，按 `output-contract.md` 的首轮输出契约重新组织结果。
4. 不得在这类结果里混入“已解决的问题 / 仍未解决的问题 / 新增副作用”等复检字段。

## 复检流程

### 第一步：完成可比性检查

只有通过可比性检查后，才允许继续下面的复检步骤。

### 第二步：锁定上一轮基线

先把上一轮信息锁定成基线，不允许复检时丢失原问题。

基线至少包含：

- 上一轮等级
- 上一轮主问题层级
- 上一轮 1 到 3 个关键缺陷
- 上一轮修正优先级

### 第三步：重跑七层校验

对当前版本重新执行完整七层校验，不得只看局部改动。

目的：

- 防止只盯着旧问题，漏掉新问题
- 防止表面修改掩盖新的结构副作用

### 第四步：逐项比较原问题状态

上一轮每个关键缺陷都必须落入以下四类之一：

- 已解决
- 部分解决
- 仍未解决
- 形式变化但本质未变

判定规则：

#### 已解决

- 相关缺陷不再影响主目标
- 新版本提供了实质补足，而不是只换说法

#### 部分解决

- 问题有所收窄，但影响仍在
- 例如补了过程片段，但还不足以支撑核心结论

#### 仍未解决

- 结构缺口依旧存在
- 只是改了措辞，没有改变问题所在层级

#### 形式变化但本质未变

- 看起来更顺
- 但根因仍在同一层
- 常见于把空话换成更利落的空话

### 第五步：检测新增副作用

复检必须额外检查是否引入以下副作用：

- 为了更快进入，牺牲了兑现
- 为了更强结论，牺牲了边界
- 为了更像真实过程，加入了无关过程噪音
- 为了更像反常识，变成强行唱反调
- 为了更完整，节奏显著变慢
- 为了更短，论证被偷掉

新增副作用一旦出现，必须单列出来，不能混进“仍未解决”里。

### 第六步：更新等级与优先级

在完成比较后，重新给出：

- 当前版本等级
- 与上一版相比是上升、持平还是下降
- 下一轮优先级

等级更新必须基于当前版本整体状态，而不是只看某一个局部改好了没有。

## 假修复识别规则

以下情况视为“假修复”，必须明确指出：

### 1. 更顺但不更准

表现：

- 文案更流畅
- 但核心判断仍无支撑

### 2. 更狠但不更深

表现：

- 结论更刺激
- 但没有新增机制和边界

### 3. 过程更多但证据仍弱

表现：

- 加了经历描述
- 但过程并不能证明核心判断

### 4. 边界更多但承诺仍失真

表现：

- 结尾加了免责声明
- 但前面的承诺仍然过大

### 5. 结构更满但主问题更散

表现：

- 多了模块
- 但没有围绕一个核心问题收束

## 复检输出顺序

复检输出固定按以下顺序组织：

1. 版本结论
2. 已解决的问题
3. 仍未解决的问题
4. 新增副作用
5. 当前等级变化
6. 下一轮修改重点

## 复检输出要求

### 版本结论

必须回答：

- 这一版相较上一版是实质改善，还是表面改善

### 已解决的问题

只写真正解决的问题，不把“稍微变好”写成已解决。

### 仍未解决的问题

必须点明它为什么还在，不能只说“还不够”。

### 新增副作用

必须说明副作用来自哪种修改取舍。

### 当前等级变化

必须明确写出：

- 从什么等级变到什么等级
- 变化的主要原因是什么

### 下一轮修改重点

必须只保留最该优先处理的 1 到 2 项，不把复检再次变成长清单。

## 复检输出边界

真复检输出中，不再重复首轮的“主问题层级 / 关键缺陷 / 修正优先级”栏位名称。

如果必须提到这些信息，也只能放在“版本结论”或“仍未解决的问题”里，用于解释变化，不得把首轮和复检两套格式混排在一份结果中。

## 复检模式下的禁止事项

以下做法禁止出现：

- 只对新版重新点评，不回看旧问题
- 只说“有进步”，不说进步在哪里
- 把新出现的问题忽略掉
- 因为局部优化成功，就把等级整体抬高
- 把表面更顺误判为结构已修复

## 复检完成标准

只有同时满足以下条件，才算一次有效复检：

1. 上一轮关键问题都有去向
2. 新版本做过完整七层重检
3. 新增副作用被显式识别
4. 等级变化有明确依据
5. 下一轮重点足够收敛
