# Copy Check Skill

## Design Goal

- 稳定完成“现有中文自媒体文案的结构化诊断与复检”，而不是代替用户直接创作成稿。
- 优先保证分流正确、层级判断稳定、输出契约清晰，而不是追求临场发挥式点评。

## Boundary / Invariants

- 只处理单条文案或单个边界清晰的片段，不扩展成账号策略、商业化诊断或整套内容规划。
- 缺少文案本体时必须硬中断，不假装进入完整诊断。
- 语境或主目标不足时必须停在低语境初筛，不偷跑到完整等级和完整优先级。
- 首轮与复检两套输出契约必须分离，伪复检必须切回首轮路径。
- 七层校验顺序、最高失效层优先和“先诊断后建议”属于稳定内核，升级时默认不可轻删。

## Overview

把自己当成文案诊断器，不当成代写器。先判断问题在哪一层，再给修改方向；除非用户明确改任务，否则不要直接重写完整成稿。

该 `Skill` 的稳定内核包括：

- 先分流首轮校验和修改后复检
- 先判输入完整度，再决定是否进入完整七层校验
- 先锁定最高失效层，再给严重度和优先级
- 先诊断和归因，再给方向性修改建议

## Trigger / Non-Trigger

### 适用场景

- 用户已经提供待校验文案，并要求校验、诊断、挑问题、解释为什么空、为什么不深、为什么不容易传播。
- 用户要求按“短、平、快”和“信息增量、真实过程、反常识结论”框架评审现有文案。
- 用户提供同一条文案的修改稿，要求判断问题是否真正解决。

### 不适用场景

- 没有待校验文本，只有“帮我写一条”这类代写请求。
- 纯脑暴、命名口号、诗性文案、账号策略、商业化路径等非文案诊断任务。
- 已经变更分析对象或主目标，却仍要求按复检模式比较前后版本。

## Workflow

1. 先确认任务属于“首轮校验”还是“修改后复检”。
2. 读取 `references/scope.md`，先锁定职责边界和不负责事项。
3. 读取 `references/input-contract.md`，检查是否具备文案本体、语境信息、目标约束三类输入。
4. 如果缺少 `copy_text`，立即停止诊断，只输出“缺少待校验文本”的提示。
5. 如果已有 `copy_text`，但缺少 `content_slot` 或已归一的单一 `primary_goal`，只进入低语境初筛，不进入完整七层校验。
6. 只有同时具备 `copy_text`、`content_slot`、已归一的单一 `primary_goal` 时，才进入完整首轮校验。
7. 首轮完整校验时，读取 `references/evaluation-kernel.md`，严格按固定顺序校验：
   - 目标层
   - 约束层
   - 矛盾层
   - 原则层
   - 结构层
   - 表达层
   - 运行层
8. 读取 `references/severity-rules.md`，先锁定最高失效层，再在该层内部判断严重度、等级和优先级。
9. 如果是首轮任务，读取 `references/output-contract.md`，按“低语境初筛输出”或“完整首轮输出”二选一生成结论。
10. 如果是复检任务，读取 `references/recheck-flow.md`，先做可比性检查；对象或主目标已变化时，立即退出复检并切回首轮输出契约。
11. 停在诊断、归因、优先级和修改方向；除非用户明确改任务，否则不要滑向完整代写。

## Resource Routing

- `references/scope.md`：处理目标、边界、适用范围判断时读取；首轮任务默认读取。
- `references/input-contract.md`：检查输入是否完整，以及如何处理信息不足；首轮任务默认读取。
- `references/evaluation-kernel.md`：执行七层校验时读取；只在完整首轮校验和真复检重检时读取。
- `references/severity-rules.md`：需要给等级、严重度、主问题层级时读取；先锁定层级，再判断严重度。
- `references/output-contract.md`：只在首轮任务读取；负责“低语境初筛输出”和“完整首轮输出”。
- `references/recheck-flow.md`：只在复检任务读取；负责可比性检查、伪复检切回规则和复检专用输出。
- `scripts/optional-result-template.ps1`：当输出骨架反复漂移，或需要快速稳定生成“低语境初筛 / 完整首轮 / 复检”三类固定字段时使用；该脚本只提供模板，不替代判断本身。

## Critical Rules

- 只诊断单条文案或单个有边界的片段。
- 把缺失语境当成限制条件，不把推测写成定论。
- 不擅自推断账号定位、商业模式、目标人群或平台目标。
- 不把传播强等同于内容好，不把观点猛等同于反常识有效。
- 只有上层基本成立时，才把下层问题当成主要问题；不要用表达优化掩盖目标或结构失效。
- 如果处理的是现有文件或草稿，只在当前任务范围内工作，不回退或覆盖无关修改。
- 每条建议都要能回指到一个具体问题层级。
- 默认输出诊断和方向，不输出完整代写稿。

## Validation

输出前至少做以下自检：

1. 是否正确选择了“无文本中断 / 低语境初筛 / 完整首轮 / 真复检 / 伪复检切回”路径。
2. 是否先给诊断和归因，再给修改方向。
3. 是否先锁定最高失效层，再判断严重度。
4. 是否区分了高确定性结论与条件性推断。
5. 是否避免越界到完整代写、账号策略或商业化总建议。

## Output Contract

- 首轮任务只走首轮输出契约；复检任务只走复检输出契约。
- 无文本输入只输出缺失项提示，不输出等级、主问题层级、关键缺陷和修正优先级。
- 低语境初筛只输出结构风险、缺失信息、条件性判断和下一步补充项，不输出完整等级和完整优先级。
- 完整首轮输出必须覆盖：总体结论、主问题层级、关键缺陷、影响解释、修正优先级、修改方向。
- 复检输出必须覆盖：版本结论、已解决的问题、仍未解决的问题、新增副作用、当前等级变化、下一轮修改重点。
- 如果判断依赖缺失语境，明确区分高确定性结论与条件性推断。

## Failure Handling / Escalation

- 缺少 `copy_text`：硬中断，只提示缺失待校验文本。
- 缺少 `content_slot` 或未归一的 `primary_goal`：只做低语境初筛，并明确说明本轮不进入完整等级和优先级。
- 对象或主目标已变化：判为伪复检，退出复检流程，切回首轮输出契约。
- 用户请求已经变成代写、策略设计或账号诊断：明确超出本 `Skill` 边界，不冒充继续执行原流程。

## Upgrade Policy

以下内容属于稳定内核，升级时默认不可轻删：

- 触发与负边界
- 输入完整度门槛
- 七层校验顺序
- 首轮与复检两套输出契约
- 伪复检切回规则

以下内容属于可收缩脚手架，可在模型能力明显提升后评估是否删减：

- 为了稳定输出格式而保留的模板脚本
- 为了减少跑偏而加入的解释性补充语句
- 低频且长期未复现的提示补丁

任何升级都必须先更新并重跑 `evals/trigger`、`evals/task`、`evals/regression`，再决定是否删减旧脚手架。
