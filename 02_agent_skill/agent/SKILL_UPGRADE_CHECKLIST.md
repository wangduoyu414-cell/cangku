# 正式 `Skill`升级检查清单

## 1. 使用时机

以下场景 `SHOULD`使用本清单：

- 模型版本升级。
- `description`改写。
- 主流程重构。
- 工具契约调整。
- 关键脚本替换。
- 正式发布前复核。

## 2. 升级前检查

- 旧版核心 `evals`结果是否已冻结。
- 当前 `Skill`是否明确了长期资产与临时脚手架。
- `skill.yaml`中的 `upgrade_eval_suite`是否齐全。
- 有副作用的路径是否已有安全评测。

## 3. 升级中检查

- 新模型是否通过旧版核心评测。
- 是否出现新的误触发。
- 是否出现新的工具误用。
- 是否出现新的安全违规。
- 是否出现新的高成本路径。

## 4. 删减检查

候选删减对象是否满足以下条件：

- 主要用于弥补旧模型基础推理不足。
- 删除后核心评测无显著退化。
- 可由工具契约、脚本或评测替代。
- 连续多轮模型都不再需要。

若上述条件不满足，则 `SHOULD NOT`删除。

## 5. 不可删检查

以下内容是否被误删：

- 触发边界。
- 负边界。
- 输出契约。
- 风险升级点。
- 安全规则。
- 工具副作用说明。
- 关键验证逻辑。
- 评测资产。

## 6. 补规则检查

新增规则前先确认：

- 是否确实对应高频失败模式。
- 是否能被更好的工具契约替代。
- 是否能被脚本替代。
- 是否会与既有规则冲突。
- 是否会让正文变成历史补丁堆。

## 7. 发布前最终检查

- `SKILL.md`与 `skill.yaml`是否一致。
- `evals`是否全部通过。
- 升级理由是否可追溯。
- 删减与新增规则是否有记录。
- 是否仍与设计目标、结构设计、验收标准一致。
