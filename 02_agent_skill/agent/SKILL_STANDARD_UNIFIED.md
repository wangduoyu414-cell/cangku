# 统一 `Skill` 标准与执行计划

## 1. 文档目标

本文档用于把两套能力合并为一条统一标准：

- 保留 `skill-creator` 的轻量创建路径、渐进加载和低成本起步方式。
- 吸收现有正式规范中的机器字段、评测门禁、升级治理和正式发布约束。

本文档回答六个问题：

- 一个 `Skill` 从草稿到正式发布应经过哪些阶段。
- `description`、`SKILL.md`、`skill.yaml`、`evals/` 分别负责什么。
- 哪些目录是正式能力必须具备的，哪些目录按需补充。
- 什么情况下可以称为“正式 `Skill`”。
- 模型升级后，哪些内容可以删，哪些内容不能删。
- 仓库当前应按什么顺序完成统一收口。

## 2. 适用范围

本文档适用于本仓库内所有新增或升级的 `Skill`。

本文档覆盖：

- `Skill` 包结构
- 触发语义
- 正文结构
- 资源路由
- 机器可读元数据
- 评测门禁
- 升级治理
- 迁移执行顺序

本文档不覆盖：

- 全局 `AGENTS.md` 语气、格式和语言规则
- 平台调度器、注册中心、存储系统的实现细节
- `Tool` 或 `MCP` 的内部算法实现
- `Agent` 编排、权限隔离和运行时状态模型

平台运行时契约仍以 `RUNTIME_CONTRACT_SPEC.md` 为准。

## 3. 非目标

本文档不是：

- 一个新的大而全平台母规范
- 对旧文档的逐段复制
- 只服务当前模型弱点的提示补丁集合
- 一个只讲理念、不提供执行门禁的设计稿

## 4. 设计目标声明

### 4.1 主目标

建立一套统一的 `Skill` 生命周期标准，使一个 `Skill` 同时具备以下能力：

- 易创建
- 易触发
- 易执行
- 易验证
- 易升级
- 易治理

### 4.2 目标优先级

各项目标的优先级固定如下，低优先级目标不得破坏高优先级目标：

1. 设计目标一致
2. 触发准确与边界清晰
3. 执行稳定与验证清晰
4. 可演进而不失控
5. 上下文效率与成本效率

### 4.3 统一设计原则

- 先定义职责边界，再写正文。
- 先定义完成标准，再定义工作流。
- 先定义不变量，再决定哪些脚手架可删。
- 先定义评测门禁，再允许对外称“正式能力”。
- 先比较升级前后证据，再修改规则。

## 5. 边界对齐

### 5.1 `description`

`description` 只负责：

- 这个 `Skill` 做什么
- 什么时候应触发
- 什么输入、文件、产物、场景是强信号
- 哪些相邻场景不应触发

`description` 不负责：

- 触发后的详细流程
- 大段领域知识
- 工具参数细节
- 发布门禁解释

### 5.2 `SKILL.md`

`SKILL.md` 只负责：

- 触发后的默认执行路径
- 资源路由
- 关键规则
- 输出契约
- 失败信号与升级处理
- 升级入口

`SKILL.md` 不负责：

- 机器治理字段
- 平台注册字段
- 冗长背景知识的长期堆积

### 5.3 `skill.yaml`

`skill.yaml` 只负责：

- 机器可读身份
- 风险级别
- 所需工具
- 冲突关系
- 默认评测集
- 升级评测集
- 兼容性和脚手架删减阈值

`skill.yaml` 不负责：

- 复述正文
- 复制 `description`
- 承载长篇自然语言流程

### 5.4 `references/`

用于放按需读取的长文档、规则说明、接口说明、领域知识。

### 5.5 `scripts/`

用于放高确定性、重复度高、长期手写成本高的逻辑。

### 5.6 `assets/`

用于放模板、骨架、静态资源和输出用素材。

### 5.7 `examples/`

用于放高代表性短样例，支持作者理解和评测输入对齐。

### 5.8 `evals/`

用于放触发、任务、回归、安全和升级评测资产。

## 6. 不变量

以下规则属于统一标准的不变量：

1. `description` 永远是首触发入口，触发语义不得隐藏在正文中。
2. `SKILL.md` 永远是触发后的执行说明，不承担机器治理角色。
3. `skill.yaml` 永远短小、结构化、可比较，不写长篇说明。
4. 资源存在但正文未显式指路，视为设计缺陷。
5. 正式 `Skill` 必须带最小 `examples/` 与最小 `evals/`。
6. 升级必须先冻结旧基线，再删减脚手架，再跑回归。
7. 触发边界、输出契约、风险升级点、关键评测资产默认不可轻删。
8. 确定性重复逻辑优先下沉到 `scripts/`，而不是长期留在正文里反复解释。

## 7. 统一生命周期

统一生命周期分四级。

### 7.1 `draft`

目标：先产出一个可触发、可执行的最小能力。

最低要求：

- `SKILL.md`
- `name`
- `description`
- `agents/openai.yaml`

通过条件：

- 能明确说明使用场景与负边界
- 能在触发后给出默认路径
- 可通过基础结构校验

### 7.2 `candidate`

目标：从“能用”升级为“可治理、可初步验证”。

最低要求：

- `draft` 全部要求
- `skill.yaml`
- `examples/`
- `evals/trigger`
- `evals/task`

通过条件：

- 有应触发和不应触发样本
- 有标准、边界、易错任务样本
- `description`、`SKILL.md`、`skill.yaml` 一致

### 7.3 `formal`

目标：成为正式发布和回归治理基线。

最低要求：

- `candidate` 全部要求
- `evals/regression`
- 非只读 `Skill` 需要 `evals/safety`
- 升级相关字段

通过条件：

- 通过正式评测门禁
- 关键字段完整
- 风险边界与副作用语义明确

### 7.4 `upgrade`

目标：在模型升级或规则重构后安全删减和验证。

最低要求：

- 冻结旧版基线
- 标记候选删减项
- 运行升级回归
- 记录删减与新增理由

通过条件：

- 无显著退化
- 未误删不可删内容
- 新旧失败模式差异已记录

## 8. 标准包结构

### 8.1 正式能力的推荐结构

```text
skill-name/
├── SKILL.md
├── skill.yaml
├── agents/
│   └── openai.yaml
├── examples/
├── evals/
├── references/          # optional
├── scripts/             # optional
└── assets/              # optional
```

### 8.2 必选目录

- `SKILL.md`
- `skill.yaml`
- `agents/openai.yaml`
- `examples/`
- `evals/`

### 8.3 按需目录

- `references/`
- `scripts/`
- `assets/`

### 8.4 设计规则

- `draft` 允许先缺 `skill.yaml` 和完整 `evals/`。
- `formal` 不允许缺 `skill.yaml`、`examples/`、`evals/`。
- 目录存在必须有清晰职责，不能为“将来可能用到”提前占位。

## 9. 头部、正文与机器字段的统一职责

### 9.1 头部元数据

统一要求：

- `name` 稳定、唯一、简短
- `description` 覆盖做什么、何时用、何时不用、强信号、负边界

### 9.2 正文模板

统一正文默认模板如下：

- `Design Goal`
- `Boundary / Invariants`
- `Overview`
- `Trigger / Non-Trigger`
- `Workflow`
- `Resource Routing`
- `Critical Rules`
- `Validation`
- `Output Contract`
- `Failure Handling / Escalation`
- `Upgrade Policy`

### 9.3 机器字段最小集

正式 `skill.yaml` 至少包含：

- `id`
- `version`
- `owner`
- `mode`
- `side_effect`
- `requires_tools`
- `preferred_over`
- `conflicts_with`
- `eval_suite`

若进入升级治理，还应补：

- `capability_floor`
- `capability_target`
- `scaffolding_removable_above`
- `upgrade_eval_suite`

## 10. 资源路由统一规则

统一要求：

- 只有在触发后才读取长参考资料
- 只有在需要确定性执行时才调用脚本
- 只有在输出需要时才使用模板和素材
- 每个目录的读取条件必须在正文中写明

推荐写法：

- 什么场景读哪份 `references/` 文件
- 什么场景跑哪一个 `scripts/` 入口
- 什么场景复用哪一个 `assets/` 模板
- 什么场景进入 `evals/` 做验证或回归

## 11. 统一评测门禁

### 11.1 基础门禁

所有阶段至少应通过：

- 基础结构校验
- `description` 与正文一致性检查
- `agents/openai.yaml` 与正文一致性检查

### 11.1.1 严格门禁适用范围

以下对象应进入严格门禁，而不是只停留在基础结构校验：

- 所有 `formal` 级 `Skill`
- 所有声称具备升级治理能力的 `Skill`
- 所有带 `skill.yaml`、`examples/`、`evals/` 的正式候选发布包

严格门禁至少额外检查：

- `SKILL.md` 是否显式写出设计目标、边界和不变量
- 统一正文关键章节是否齐全
- 是否仍残留占位符或未完成项
- `examples/` 与 `evals/` 是否具备最小结构
- `upgrade_eval_suite` 与升级资产是否一致

### 11.2 `candidate` 门禁

至少覆盖：

- 一组应触发样本
- 一组不应触发样本
- 三类任务样本：标准、边界、易错

### 11.3 `formal` 门禁

至少覆盖：

- `trigger`
- `task`
- `regression`

当 `side_effect` 不是 `read_only` 时，必须再覆盖：

- `safety`

### 11.4 升级门禁

至少覆盖：

- 冻结旧版核心结果
- 跑新模型核心评测
- 进行删减前后对比
- 跑升级回归

### 11.5 统一验收视角

正式验收固定从四个视角检查：

- `Outcome`
- `Process`
- `Style`
- `Efficiency`

## 12. 统一升级治理

### 12.1 升级前

必须先完成：

- 冻结旧版核心 `evals` 结果
- 标记长期资产与临时脚手架
- 确认 `upgrade_eval_suite` 齐全
- 对非只读路径确认安全评测存在

### 12.2 升级中

必须按以下顺序执行：

1. 运行旧版核心评测
2. 在新模型上跑核心集
3. 标记新增失败模式
4. 标记候选删减项
5. 删除候选脚手架
6. 重跑 `upgrade` 与 `regression`
7. 比较触发、任务、误用、安全、成本

### 12.3 升级后

必须记录：

- 删除了哪些旧脚手架
- 为什么可以删
- 保留了哪些不可删内容
- 新增了哪些规则
- 新规则对应什么高频失败模式

## 13. 作者工作流

作者统一按以下顺序工作：

1. 先明确设计目标、非目标和职责边界
2. 收集真实使用样例
3. 先写 `description`
4. 再写 `SKILL.md` 的默认路径
5. 决定哪些知识下沉到 `references/`
6. 决定哪些逻辑下沉到 `scripts/`
7. 补 `skill.yaml`
8. 写最小 `examples/`
9. 写最小 `evals/`
10. 跑基础校验
11. 进入阶段门禁
12. 达标后再提升生命周期等级

## 14. 仓库执行计划

### 14.1 第一阶段：统一入口

动作：

- 新增本文件作为 `Skill` 统一入口
- 更新 `README.md`
- 停止把 `SKILL_DESIGN.md` 作为唯一入口

完成定义：

- 仓库内只保留一个技能设计主入口

### 14.2 第二阶段：子规范归位

动作：

- 保留 `SKILL_MANIFEST_SPEC.md`
- 保留 `SKILL_EVALS_SPEC.md`
- 保留 `SKILL_UPGRADE_CHECKLIST.md`
- 将它们视为本文件的子规范，而不是并列母规范

完成定义：

- 每类规则只有一个权威归属

### 14.3 第三阶段：试点迁移

动作：

- 选择一个现有 `Skill` 做试点迁移
- 推荐优先迁移 `copy-check-skill`
- 对照本文件检查缺口

完成定义：

- 至少一个现有 `Skill` 可被明确判定为 `draft`、`candidate` 或 `formal`

### 14.4 第四阶段：门禁固化

动作：

- 把生命周期门禁写入作者流程
- 为正式能力补齐最小评测清单
- 为升级能力补齐升级回归清单

完成定义：

- 新增 `Skill` 不再依赖临场解释决定是否可发布

### 14.5 第五阶段：升级治理落地

动作：

- 形成升级前冻结、升级后比对的固定流程
- 对高价值 `Skill` 补兼容性字段和升级评测集

完成定义：

- 模型升级后可以先删脚手架，再用证据证明没有退化

## 15. 完成定义

一个 `Skill` 只有在满足以下条件后，才可称为“正式能力”：

- 已达到 `formal` 生命周期等级
- `description` 与负边界清晰
- `SKILL.md` 有默认路径、资源路由、输出契约和失败升级
- `skill.yaml` 字段完整
- 具备最小 `examples/`
- 具备正式门禁要求的 `evals/`
- 副作用语义明确
- 升级规则明确

## 16. 迁移判断规则

若一个现有 `Skill`：

- 只有 `SKILL.md`，判为 `draft`
- 有 `skill.yaml` 和最小评测，但缺正式回归，判为 `candidate`
- 有正式回归和安全门禁，判为 `formal`
- 已有冻结基线和删减验证，判为 `upgrade-ready`

## 17. 结语

统一标准的目标，不是把 `Skill` 写得更重，而是把真正影响长期质量的对象稳定下来：

- 触发语义放在头部
- 方法收束放在正文
- 机器治理放在 `skill.yaml`
- 长知识下沉到 `references/`
- 确定性动作下沉到 `scripts/`
- 升级判断交给 `evals`

轻量创建保留，正式治理补齐，升级删减受控。
