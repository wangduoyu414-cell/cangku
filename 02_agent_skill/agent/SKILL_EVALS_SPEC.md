# 正式 `evals`设计与升级规范

## 1. 设计目标

`evals`不是附属材料，而是 `Skill`正式治理的一部分。  
没有评测，升级治理就会退化为主观判断。

目标：

- 触发准确率可度量。
- 任务完成质量可回归。
- 工具误用可定位。
- 安全退化可发现。
- 模型升级后的删减动作有证据支撑。

## 2. 基本原则

- `evals` `MUST`覆盖真实任务，而不只是演示样例。
- `evals` `MUST`与主设计目标一致。
- `evals` `SHOULD`可独立运行、可比较、可回归。
- 模型升级前后 `MUST`优先比较评测结果，而不是优先改正文。

## 3. 推荐目录

正式 `Skill` `SHOULD`按类拆分：

- `evals/trigger/`
- `evals/task/`
- `evals/regression/`
- `evals/safety/`
- `evals/upgrade/`

## 4. 各类评测职责

### 4.1 `trigger`

验证：

- 该触发时是否触发。
- 不该触发时是否安静。
- 相邻 `Skill`是否边界混乱。

### 4.2 `task`

验证：

- 主路径是否完成。
- 边界路径是否可处理。
- 易错路径是否稳定。

### 4.3 `regression`

验证：

- 规则修改后是否退化。
- 脚本替换后是否行为漂移。
- 输出格式是否被破坏。

### 4.4 `safety`

验证：

- 是否越权。
- 是否被注入式输入诱导。
- 是否错误扩大作用范围。
- 是否误用高风险工具。

### 4.5 `upgrade`

验证：

- 模型升级后原有基线是否仍成立。
- 旧脚手架删减后是否退化。
- 新模型是否暴露出新的失败模式。

## 5. 最小覆盖要求

正式 `Skill`最少应包含：

- 一组 `trigger`正样本。
- 一组 `trigger`负样本。
- 三类 `task`样本：标准、边界、易错。
- 一组 `regression`核心集。

若 `side_effect`不是 `read_only`，还 `SHOULD`包含：

- 一组 `safety`评测。
- 一组 `upgrade`删减验证。

## 6. 单条评测样本应包含的内容

每条正式样本 `SHOULD`至少包含：

- `id`
- `goal`
- `input`
- `expected`
- `type`
- `notes`

如果是工具型场景，还 `SHOULD`补：

- `expected_tools`
- `forbidden_tools`
- `expected_side_effect`

## 7. 升级评测流程

模型升级时，`upgrade`评测 `SHOULD`按以下顺序执行：

1. 固定旧模型基线结果。
2. 运行新模型全量核心集。
3. 标记新增失败模式。
4. 尝试删除候选旧脚手架。
5. 重新运行 `upgrade`与 `regression`集。
6. 只在删减失败或新增失败模式显著时补新规则。

## 8. 删减验证规则

删减旧脚手架前后，至少比较以下指标：

- 触发精确率。
- 触发召回率。
- 任务通过率。
- 工具误用率。
- 安全违规率。
- 成本与时延。

若删减后无显著退化，则该脚手架 `SHOULD`移除。  
若删减后退化明显，则该脚手架仍有当前价值，不应草率删除。

## 9. 验收门槛

正式 `evals`验收至少检查：

- 样本是否覆盖真实任务。
- 是否覆盖应触发与不应触发。
- 是否覆盖标准、边界、易错路径。
- 是否具备升级前后可比较性。
- 是否与 `skill.yaml`中的 `eval_suite`声明一致。
