# Implementation Report Template

Fill this template after implementation, or use it as a stop report when the task cannot proceed safely.

> **Schema note**: Each section header includes an anchor tag (`##-Anchor-Name`) for programmatic parsing.
> The Validation Distinction section is machine-checked - never falsify it.
>
> **Evidence note**: When a command actually runs, prefer binding the report to a tool-generated
> execution receipt instead of relying on free-form prose alone. Keep the prose short and use the
> receipt fields to anchor what really happened.
>
> For `minimal` depth:
> - `Covered Edge Cases` may contain a single concise item
> - `Residual Risks` may contain a single concise item
> - `Executed Validation` and `Validation Not Run` may use compact one-item forms
> - `Validation Distinction` remains fully required

## Strict Validation Rules

### Validation Distinction is Non-Negotiable

The **Validation Distinction** section is the most critical - and most frequently violated - part
of this template. The following rules are absolute:

1. **Only report validation that actually ran.** Do not claim a test passed if you did not
   run it, did not see its output, or are inferring its result from context.

2. **Executed Validation must match Validation Not Run.** If you list something in
   "Validation Not Run," you cannot also list it in "Executed Validation."

3. **Commands must be specific.** "Run the tests" is not acceptable. You must provide
   the actual command (for example, `pytest tests/test_parser.py -v`).

4. **Results must be real.** Do not write "passed" or "success" without evidence.
   At minimum, describe what the output showed.

5. **Prefer receipt-backed evidence.** If a command was executed, include at least
   an exit code and a minimal evidence digest; if a tool-generated receipt exists,
   reference it.

5. **FATAL violation: claiming un-run validation as executed.** If `claimed_verified_count`
   exceeds `actually_verified_count`, the report is invalid.

### Stop Conditions

Use the stop report template when:

1. Required input shape or enum semantics are unknown and guessing would change behavior.
2. Return contract would be silently changed by default/fallback.
3. Side effects cross file, service, or persistence boundaries without approval.
4. The task expands beyond local implementation quality into broader design decisions.
5. Fallback logic would change business meaning.

### Report Completeness Checklist

- [ ] **Report Metadata** - generated_at, task_mode, continuation_mode, and implementation state are specified
- [ ] **Contract Depth** - contract_depth is specified as minimal / standard / expanded
- [ ] **Language-Specific Rules Applied** - language and prohibited patterns status are both specified
- [ ] **Covered Edge Cases** - `minimal` allows a concise entry; `standard` / `expanded` expect detailed entries
- [ ] **Residual Risks** - `minimal` allows a concise entry; `standard` / `expanded` expect detailed entries
- [ ] **Executed Validation** - `minimal` allows a concise entry; `standard` / `expanded` expect detailed entries
- [ ] **Validation Not Run** - `minimal` allows a concise entry; `standard` / `expanded` expect detailed entries
- [ ] **Validation Distinction** - explicitly lists `none` or actual un-run validation; discrepancy must be 0
- [ ] **Contract Deviations** - explicitly states `None` or describes deviations
