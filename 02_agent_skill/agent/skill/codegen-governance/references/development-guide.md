# Codegen Execution Constraint 提示开发指导手册

版本：v0.1
定位：后续开发与迭代提示词时的统一指导手册
适用对象：Skill 设计者、Prompt 维护者、Agent 产品工程师、质量治理设计者

---

## 1. 文档目标

本文档用于指导后续如何开发、扩展、评审和迭代 `codegen-governance` 这类"代码生成前硬约束"提示体系。

本文档关注的不是"如何写代码"，而是"如何在写代码前用提示词把风险边界收紧"。

这份手册的目标有四个：

1. 让提示体系长期保持统一口径。
2. 让新增场景时知道应该往哪里加、怎么加。
3. 让不同维护者写出来的 prompt 模块风格一致。
4. 让提示体系可以被评测、被审计、被持续演进。

---

## 1.1 维护入口

在新增或调整 `references/` 下的文件之前，先读这两个文件：

1. `references/durable-reference-index.md`
2. `references/reference-governance.md`

它们分别解决两个问题：

1. 某条内容应该归到哪一类长期资产里。
2. 什么内容可以提升为长期规则，什么内容应停留在样例、fixture 或历史文档里。

维护时优先遵守以下顺序：

1. 先判断内容属于长期规范、环境层指导、场景包、样例/fixture，还是历史/路线图。
2. 再决定它应该放在哪个目录，而不是先写文件再找位置。
3. 如果内容主要是 repo-specific、版本敏感或仅用于回归验证，不要直接提升到长期规则层。

## 2. 设计定位

`codegen-governance` 的本质不是工作流，也不是代码修复器。

它是一个"生成前约束层"，职责是：

1. 在代码生成前识别本次任务的最小实现风险。
2. 强制模型先产出 `Pre-Generation Contract`。
3. 在契约不完整时阻止模型直接开始写代码。
4. 在写完代码后强制回填真实验证和剩余风险。

它不负责：

1. 仓库级设计评审。
2. 大范围架构改造建议。
3. 后置 code review。
4. 自动修复所有实现问题。

一句话定义：

> 它是代码生成前的边界收束器，而不是代码生成器本身。

---

## 3. 适用范围

适合触发的任务：

1. 补函数实现。
2. 修局部 bug。
3. 增加一个分支或 fallback。
4. 修改返回结构或错误处理。
5. 新增缓存、重试、幂等、去重逻辑。
6. 改动异步、时序、并发、资源释放。
7. 处理输入解析、字段映射、归一化、协议转换。
8. 添加日志、埋点、审计输出。

不适合直接由本 skill 主导的任务：

1. 需求尚未澄清。
2. 目标是系统设计或模块拆分。
3. 目标是仓库级重构。
4. 目标是对现有变更做纯 review。
5. 任务本身不生成代码。

## 4. 核心设计原则

后续所有 prompt 开发都必须围绕以下原则展开。

### 4.1 先收束边界，再生成代码

提示词必须先让模型确认：

1. 要改哪里。
2. 允许改多大。
3. 哪些语义不能变。
4. 哪些失败路径必须显式处理。
5. 哪些验证是真能做、哪些现在做不了。

如果这些信息不清楚，提示词必须让模型先停下来，而不是直接进入实现。

### 4.2 优先保护局部实现质量

提示体系的重点是"最小实现是否可靠"，不是"顺手把整个模块做漂亮"。

因此后续新增 prompt 时必须优先约束：

1. 输入是否合法。
2. 分支是否完备。
3. 默认值是否掩盖错误。
4. 返回语义是否稳定。
5. 副作用是否显式。
6. 并发和重试是否可控。
7. 边界协议和日志是否真实可靠。

### 4.3 显式优于隐式

提示词应持续把模型往这三个方向推：

1. 显式拒绝。
2. 显式降级。
3. 显式说明不确定性。

不要鼓励模型靠：

1. 宽松解析。
2. 隐式类型转换。
3. 宿主默认行为。
4. 模糊返回值。
5. 静默 fallback。

### 4.4 契约先于实现

后续所有 prompt 模块都要服务于一个核心产物：

`Pre-Generation Contract`

如果一个新增 prompt 模块不能帮助契约更完整、更可检查，通常不应加入主流程。

### 4.5 真实验证优先

提示词必须长期压制"虚报验证"的倾向。

无论后续加什么模块，都要保留这条总原则：

1. 只报告真实执行的验证。
2. 没跑的测试不能写成已通过。
3. 没确认的风险不能伪装成已覆盖。

## 5. 提示体系总架构

建议长期维持"五层结构"，不要把所有内容堆在一个长 prompt 里。

### 5.1 第一层：基础总则层

职责：

1. 统一所有任务共享的硬约束。
2. 定义默认行为。
3. 定义禁止动作。

适合承载的内容：

1. 最小改动原则。
2. 不得混淆空值与零值。
3. 不得只写 happy path。
4. 不得隐藏副作用。
5. 不得虚报验证。

对应资源：

- `references/base-rules.md`

### 5.2 第二层：场景选择层

职责：

1. 根据任务风险选 1 到 4 个场景包。
2. 决定后续要加载哪些细化 prompt。

这个层只负责"选什么"，不负责重复解释所有细节。

对应资源：

- `references/scenario-selection.md`

### 5.3 第三层：场景约束层

职责：

1. 针对具体问题补充硬约束。
2. 约束模型在某类问题上的默认错误倾向。

当前建议保持四大场景包：

1. 输入与分支。
2. 默认值与结果契约。
3. 副作用与运行时。
4. 边界与可观测性。

对应资源：

- `references/scenario-input-branching.md`
- `references/scenario-fallback-contract.md`
- `references/scenario-side-effect-runtime.md`
- `references/scenario-boundary-observability.md`

### 5.4 第四层：契约输出层

职责：

1. 强制模型先输出可检查的实现前契约。
2. 把风险与假设结构化。

对应资源：

- `references/pre-generation-contract-template.md`

### 5.5 第五层：实施后回填层

职责：

1. 约束模型在实现后如实报告。
2. 显式记录剩余风险和未执行验证。

对应资源：

- `references/implementation-report-template.md`

## 6. 标准提示拼装顺序

后续开发 prompt 时，建议始终遵守以下拼装顺序：

1. 任务锚点恢复。
2. 基础总则。
3. 场景选择。
4. 场景约束。
5. 输出 `Pre-Generation Contract`。
6. 判断是否允许进入实现。
7. 实现后输出 `Implementation Report`。

不要把顺序反过来。

尤其不要：

1. 先写代码，再补契约。
2. 先猜输入语义，再补询问。
3. 先扩大改动范围，再解释原因。

## 7. Prompt 模块开发规范

后续新增或修改 prompt 模块时，建议统一按以下结构编写。

### 7.1 模块最小结构

每个模块应尽量包含：

1. 该模块解决什么风险。
2. 该模块适用于什么任务。
3. 该模块明确禁止什么错误行为。
4. 该模块要求模型在输出中说明什么。

推荐模板：

```text
[场景：模块名]
当你处理 X 类问题时，必须遵守：

1. ...
2. ...
3. ...

你必须说明：
- ...
- ...
- ...
```

### 7.2 模块写作原则

应该：

1. 用祈使句。
2. 先写动作，再写原因。
3. 约束真实高频错误，不要泛泛而谈。
4. 让规则可落地、可判断、可检查。

不应该：

1. 写成长篇哲学说明。
2. 混入多个不相干风险域。
3. 使用含糊词，如"尽量""最好""适当"而没有边界。
4. 只告诉模型"注意风险"，却不说具体怎么做。

### 7.3 新模块加入的门槛

只有在同时满足以下条件时，才建议新增模块：

1. 当前已有模块无法稳定覆盖该风险。
2. 该风险在真实任务中会重复出现。
3. 该风险具备清晰边界，不是模糊的大类。
4. 新模块不会和已有模块产生严重边界冲突。

## 8. 标准执行骨架

后续开发 prompt 时，建议所有组合都围绕以下执行骨架：

```text
你当前处于"代码生成前硬约束"模式。

先不要直接写代码。

先完成以下步骤：
1. 恢复本次实现锚点：目标文件、目标函数、目标改动、允许影响范围。
2. 加载基础总则。
3. 根据任务选择最相关的场景约束。
4. 输出一份 Pre-Generation Contract。
5. 如果契约信息缺失且猜测会改变行为，停止实现并明确指出缺失项。
6. 只有在契约完整后，才开始给出最小代码改动方案。
7. 实现后必须输出 Implementation Report，只能报告真实执行过的验证结果。
```

这段骨架不一定原封不动复用，但语义不要丢。

## 9. 关键 Prompt 模板

下面这几段建议作为后续开发时的标准素材库。

### 9.1 任务锚点恢复模板

```text
在开始实现前，先恢复本次任务的实现锚点，并明确输出：
- 目标文件或目标代码位置
- 目标函数、模块或逻辑片段
- 用户明确要求的变更
- 默认不允许改动的外围范围
- 任何会导致语义变化的不确定点

如果缺少这些信息且猜测会改变行为，停止实现并指出最小缺失项。
```

### 9.2 场景选择模板

```text
基于当前任务，只选择 1 到 4 个最相关的风险场景。
优先覆盖真正决定实现正确性的风险，而不是把所有场景都堆上来。

你必须说明：
- 选了哪些场景
- 每个场景为什么适用
- 哪些看起来相关但本次没有选，以及为什么不选
```

### 9.3 生成前契约模板

```text
在写代码前，必须先输出一份 Pre-Generation Contract，至少包含：
- Scope
- Selected Scenario Packs
- Input Contract
- Branching And Ordering
- Defaults And Fallback
- Return And Failure Contract
- Side Effects And Runtime
- Boundary And Observability
- Validation Plan
- Open Risks

如果任何关键项无法确定，必须在 Open Risks 中明确写出，而不是靠猜测补默认值。
```

### 9.4 停止条件模板

```text
如果出现以下任一情况，不要继续写代码，先停止并说明原因：
- 输入形状不明确
- 未知枚举值或错误语义不明确
- 返回契约可能被静默改变
- fallback 会改变业务含义
- 涉及副作用、缓存、并发或资源生命周期，但边界不清晰
- 外部协议、序列化、安全转义或日志语义不明确
```

### 9.5 实施后报告模板

```text
实现完成后，必须输出 Implementation Report，包含：
- Covered Edge Cases
- Residual Risks
- Executed Validation
- Validation Not Run
- Contract Deviations

不要报告未真实执行的验证。
```

## 10. 场景包开发指南

如果后续要继续扩展场景包，建议按以下方式规划。

### 10.1 输入与分支类

适合继续扩展的子问题：

1. 输入合法化。
2. 枚举闭包。
3. 排序稳定性。
4. 集合聚合边界。
5. 归一化时机。

新增时要重点约束：

1. 宽松解析。
2. truthy/falsy 误判。
3. 分支顺序偷带业务优先级。
4. 先格式化再比较。
5. 比较器不稳定。

### 10.2 默认值与契约类

适合继续扩展的子问题：

1. fallback 透明性。
2. 部分成功语义。
3. 旧接口兼容。
4. 错误包装和恢复。

新增时要重点约束：

1. `||` 误伤合法零值。
2. 空结果和失败复用同一返回值。
3. 抛错和返回错误对象混用。
4. 自动降级但调用方无感知。

### 10.3 副作用与运行时类

适合继续扩展的子问题：

1. 资源成对释放。
2. 超时与取消。
3. 竞态与重入。
4. 幂等键设计。
5. 重试上限与退避。

新增时要重点约束：

1. 原地修改共享对象。
2. 隐藏副作用。
3. 无上限重试。
4. 伪幂等。
5. 中途失败不收尾。

### 10.4 边界与可观测性类

适合继续扩展的子问题：

1. 时区与时间单位。
2. 金额与精度。
3. Unicode 与编码。
4. 序列化与协议转换。
5. 安全转义。
6. 日志脱敏。

新增时要重点约束：

1. 依赖宿主默认行为。
2. 显示语义与存储语义混用。
3. 外部字段默默透传。
4. 调试逻辑残留。
5. 审计记录和真实结果不一致。

---

### 10.5 高频子场景卡片格式

当不再外扩语言主栈、而是继续提升常见能力覆盖时，优先在现有四大场景包内部增加"高频子场景卡片"，而不是继续新增大包。

每张卡片建议固定为四段：

1. **When to select**：什么任务形状下必须选它。
2. **Contract questions you must answer**：进入契约前必须回答哪些问题。
3. **Common failure modes**：模型最容易踩的坑。
4. **Minimum validation**：最小可执行验证和必须承认的未验证项。

卡片设计约束：

1. 必须落入现有 A/B/C/D 四大场景包之一。
2. 必须是跨语言高频问题，而不是某个框架的私有技巧。
3. 必须能映射到 Pre-Generation Contract 或 Implementation Report 的现有字段。
4. 必须能配套新增评测样例，不能只停留在经验描述。

第一批优先补充的高频子场景建议是：

1. patch / partial update semantics
2. outbound http contract
3. cache / idempotency / duplicate suppression

第二批优先补充的高频子场景建议是：

1. time / money / precision
2. file import / export
3. audit redaction / stable diagnostics

第三批优先补充的高频子场景建议是：

1. batch partial success
2. async worker / webhook delivery

## 11. 输出契约开发规范

后续修改提示词时，绝对不要轻易放松输出契约。

标准输出至少应分两段：

1. 生成前契约。
2. 实施后报告。

### 11.1 生成前契约必须回答的问题

1. 改动范围是什么。
2. 哪些语义不能变。
3. 哪些场景包被选中。
4. 输入如何判定合法与非法。
5. 默认值和 fallback 怎么触发。
6. 返回和失败语义是什么。
7. 会不会产生副作用。
8. 有哪些并发、缓存、重试、协议、日志边界。
9. 怎么验证。
10. 还有哪些未知风险。

### 11.2 实施后报告必须回答的问题

1. 覆盖了哪些边界条件。
2. 还有哪些风险没覆盖。
3. 实际做了哪些验证。
4. 哪些验证没做。
5. 实现是否偏离了原契约。

## 12. 场景拼装指南

后续开发新 prompt 时，可以直接参考以下拼装建议。

### 12.1 修输入解析问题

建议拼装：

1. 基础总则
2. 输入与分支
3. 默认值与契约

重点关注：

1. 缺失值和非法值的区分。
2. 解析失败如何暴露。
3. 默认值是否掩盖错误。

### 12.2 修 fallback 或空值问题

建议拼装：

1. 基础总则
2. 默认值与契约

重点关注：

1. `0 / false / ''` 是否被误伤。
2. 调用方能否感知降级。
3. 返回契约是否仍稳定。

### 12.3 修异步、并发、重试问题

建议拼装：

1. 基础总则
2. 副作用与运行时
3. 默认值与契约

重点关注：

1. 超时、取消、重入。
2. 重试边界。
3. 幂等与重复副作用。

### 12.4 修外部 API、协议转换、日志问题

建议拼装：

1. 基础总则
2. 边界与可观测性
3. 输入与分支 或 默认值与契约

重点关注：

1. 边界协议是否按真实语义处理。
2. 外部字段是否被静默吞掉。
3. 日志是否泄露敏感信息。

### 12.5 修 PATCH / Partial Update 语义问题

建议拼装：

1. 基础总则
2. 输入与分支
3. 默认值与契约

重点关注：

1. 字段缺失、显式 `null`、显式空字符串、显式 `false`、显式 `0` 是否各有不同含义。
2. 缺失字段是 retain 旧值、回填默认值，还是视为非法。
3. clear / remove / replace / append 是否是不同操作。
4. 返回结果是否能让调用方知道本次到底是保留旧值、显式清空，还是写入新值。

### 12.6 修出站 HTTP / Partner API 行为问题

建议拼装：

1. 基础总则
2. 输入与分支
3. 默认值与契约
4. 副作用与运行时
5. 边界与可观测性

重点关注：

1. 请求输入、URL、headers、body、timeout 单位是否明确。
2. 哪些错误可重试，哪些错误必须直接失败。
3. 429 / 5xx / timeout / schema mismatch 是否走同一路径。
4. 外部结果如何映射为本地稳定契约。
5. 审计或日志是否会泄露敏感字段。

### 12.7 修缓存 / 幂等 / 去重问题

建议拼装：

1. 基础总则
2. 默认值与契约
3. 副作用与运行时

重点关注：

1. 幂等键或去重键的语义维度是什么。
2. 首次执行、重复执行、缓存命中、部分失败、重试中的重复请求如何区分。
3. 什么时候写入缓存或去重存储，什么时候绝不能写。
4. 调用方如何观察到 duplicate / replay / fresh success 的区别。

### 12.8 修时间 / 金额 / 精度问题

建议拼装：

1. 基础总则
2. 输入与分支
3. 边界与可观测性

重点关注：

1. canonical timezone 是什么，naive 时间是否允许。
2. 金额是 minor units、decimal string 还是 float，舍入规则是什么。
3. 存储值、比较值、展示值、导出值是否被混用。
4. 是否依赖宿主 locale、时区或浮点默认行为。

### 12.9 修文件导入导出问题

建议拼装：

1. 基础总则
2. 输入与分支
3. 副作用与运行时
4. 边界与可观测性

重点关注：

1. 文件名、路径、扩展名、列结构、编码、分隔符是否明确。
2. 文件何时创建、覆盖、追加、清理，失败后是否留下半成品。
3. 行级非法数据、未知列、坏格式是否显式拒绝或隔离。
4. 导出内容是否把真实存储语义和展示语义混在一起。

### 12.10 修审计脱敏与稳定诊断问题

建议拼装：

1. 基础总则
2. 副作用与运行时
3. 边界与可观测性

重点关注：

1. 哪些字段必须保留，哪些字段必须脱敏、哈希或省略。
2. 成功和失败路径是否遵循同一套脱敏规则。
3. 调试日志是否会越过正式审计边界。
4. 诊断信息是否足够追踪问题，但不足以泄露敏感原始内容。

### 12.11 修批处理 partial success 问题

建议拼装：

1. 基础总则
2. 输入与分支
3. 默认值与契约
4. 副作用与运行时

重点关注：

1. 单项成功、单项失败、单项跳过、整批重试、整批终止如何区分。
2. 批量汇总状态是否与每项真实结果一致。
3. 死信、失败记录、部分成功统计是在什么时候写入。
4. 调用方是否能观察到 whole success、partial success、retry-required 的差别。

### 12.12 修 webhook / worker 交付语义问题

建议拼装：

1. 基础总则
2. 输入与分支
3. 默认值与契约
4. 副作用与运行时
5. 边界与可观测性

重点关注：

1. delivery id、event id、签名验证和 provider 重试语义是否明确。
2. ack、retry、dead-letter 是按整批还是按单项发生。
3. duplicate delivery 和 duplicate event 是否使用同一套抑制语义。
4. 审计是否描述真实结果，而不是仅描述"尝试处理过"。

## 14. 常见反模式

后续维护中最容易退化成下面这些问题。

### 14.1 提示词越来越长，但边界越来越模糊

表现：

1. 什么都写了一点。
2. 但模型还是不知道第一步该做什么。

修正方式：

1. 把正文收回到流程和契约。
2. 把细节拆到场景包。

### 14.2 把约束层写成实现工作流

表现：

1. 提示词开始教模型如何搭架构、如何重构模块。
2. 逐渐脱离"最小实现质量"。

修正方式：

1. 把超出边界的内容移交更高层 workflow。

### 14.3 规则很多，但不可检查

表现：

1. 充满"注意""合理""稳妥"。
2. 但没有明确输出要求。

修正方式：

1. 每条关键规则都要对应到契约字段或报告字段。

### 14.4 只管生成前，不管生成后

表现：

1. 契约写得很好。
2. 但实现后没有真实回填。

修正方式：

1. 固定保留 `Implementation Report`。

### 14.5 为了"可运行"放松硬约束

表现：

1. 允许大量猜测。
2. 默认值无限扩张。
3. 不清楚就继续写。

修正方式：

1. 恢复停止条件。
2. 强化"显式不确定"优先级。

## 17. 语言特定约束开发指南

当新增或修改语言特定约束时，遵循以下原则：

### 17.1 语言约束文件的结构

每个语言约束文件应包含：

1. 语言惯用法（idioms）：该语言的标准做法
2. 特定陷阱（traps）：该语言独有的危险模式
3. 风格规范（style）：该语言的命名、格式化约定
4. 类型系统特性（type system）：该语言的类型相关风险

### 17.2 语言约束与通用约束的关系

语言约束**补充**通用约束，**不替代**通用约束。

- 如果语言惯用法与通用规则冲突，以语言惯用法为准。
- 如果语言陷阱在通用规则中已有覆盖，在语言文件中强化。
- 如果语言有独有的风险域，在语言文件中新增。

### 17.3 禁止模式清单的维护

禁止模式清单必须：

1. 具体到语言和版本（如 Python 3.10+）。
2. 附带正确替代方案的示例。
3. 标注违反后的典型后果。

## 18. 评测闭环体系

本 skill 已建立完整的评测闭环机制，包含三个层次、五个 stage。

### 18.1 评测架构

```
scripts/run_evals.py          # 统一入口（手动 + CI）
    ├── Stage 1: Metadata Validation   (validate_metadata.py)
    ├── Stage 2: Trigger Evaluation    (eval_trigger_and_scenario.py --trigger)
    ├── Stage 3: Scenario Selection    (eval_trigger_and_scenario.py --scenarios)
    ├── Stage 4: Task Structure        (eval_trigger_and_scenario.py --tasks)
    └── Stage 5: Contract Anchor Check (eval_contract_output.py --check-missing)
```

### 18.2 五个评测 Stage

| Stage | 名称 | 验证内容 | 对应规则 |
|-------|------|----------|----------|
| 1 | Metadata Validation | skill.yaml 与 SKILL.md frontmatter 一致性、eval_suite 完整性、模板锚点存在性、引用的文件路径有效性 | skill.yaml:26–30 |
| 2 | Trigger Evaluation | 是否正确触发 / 不触发本 skill | SKILL.md:117–136 |
| 3 | Scenario Selection | Pack A/B/C/D 选择是否正确，是否出现过选或漏选 | SKILL.md:65–90 |
| 4 | Task Structure | 任务评测用例结构是否完整 | 本章 18.3 |
| 5 | Contract Anchor Check | Pre-Generation Contract 的 MUST 锚点是否完整 | 本章 18.4 |

### 18.3 契约质量评分体系

契约质量通过 `scripts/eval_contract_output.py` 评测，评分公式为：

```
Score = (MustCompleteness × 40%) + (ShouldCompleteness × 20%)
        - max(0, (PlaceholderCount - 2) × 5)
        - max(0, (VagueFillerCount - 2) × 3)
        - (FakeValidationCount × 20)
```

质量等级：

| 等级 | 分值 | 含义 |
|------|------|------|
| A | 90–100 | 优秀契约，可进入实现 |
| B | 80–89 | 良好契约，轻微改进即可 |
| C | 70–79 | 可接受契约，需要显著改进 |
| D | 60–69 | 边缘契约，需要重大改进 |
| F | <60 | 劣质契约，不得进入实现 |

**三项绝对红线（任何一项出现即为 FATAL）**：

1. Validation Distinction 缺失（必须在报告中区分已执行 / 未执行的验证）
2. 虚报验证（声称已执行但实际未执行的测试）
3. 停止条件模板缺失（stop_before_code=true 但没有 stop 模板）

### 18.4 结构化锚点体系

契约模板和报告模板使用 Markdown 锚点标记（`##-Anchor-Name`）实现程序化解析。
这些锚点是 `eval_contract_output.py` 的检查依据，不得重命名或删除。

**契约模板锚点（按严重级别）**：

- **MUST（缺失即为 FATAL）**：Contract-Metadata、Target-Language、Scope、Selected-Scenario-Packs、Input-Contract、Return-And-Failure-Contract
- **SHOULD（缺失触发 WARN）**：Branching-And-Ordering、Defaults-And-Fallback、Side-Effects-And-Runtime、Validation-Plan
- **MAY（无惩罚）**：Boundary-And-Observability、Open-Risks

**报告模板锚点（全部为 MUST）**：
Report-Metadata、Language-Specific-Rules-Applied、Covered-Edge-Cases、Residual-Risks、Executed-Validation、Validation-Distinction（FATAL 级别）、Contract-Deviations

### 18.5 CI 集成

评测闭环必须接入 CI 才能真正生效。使用方法：

```bash
# 在 CI pipeline 中运行
python scripts/run_evals.py --ci

# 输出 JSON 格式（供 CI 系统消费）
python scripts/run_evals.py --json --output eval-results.json
```

`--ci` 模式会在第一个 stage 失败时立即退出，并在输出中提供调试信息。

建议的 CI 触发条件：
- 每次向 `main` 分支推送代码
- 每次创建 Pull Request
- 每次修改 SKILL.md、skill.yaml 或 references/ 目录

### 18.6 评测资产维护

评测资产位于 `evals/` 目录，包含：

- `trigger.json` — 触发 / 不触发判定样例（20 个应触发 + 16 个不应触发）
- `tasks.json` — 任务评测用例（26 个任务，覆盖 Python、Go、TypeScript、Vue；JavaScript 运行时规则暂复用 TypeScript）
- `scenario-selection-cases.json` — 场景选择对照样例（16 个，含正确和错误选择）
- `contract-quality-rubric.md` — 契约质量人工评分标准

维护规则：
1. 新增语言支持时，必须在 tasks.json 中增加至少 2 个该语言的任务样例
2. 新增场景包时，必须在 scenario-selection-cases.json 中增加对照样例
3. 每次修改 Pre-Generation Contract 模板后，必须在 run_evals.py 中验证锚点完整性

### 18.7 评测闭环的局限性

当前评测体系仍有以下局限，后续迭代应逐步改善：

1. **触发评估依赖关键词匹配**：eval_trigger_and_scenario.py 目前使用规则匹配判断触发，
   未来应改为用模型输出对比真实场景，以减少规则覆盖盲区。
2. **契约质量评分依赖正则匹配**：`eval_contract_output.py` 中的 placeholder 和 vague
   检测使用正则，可能出现漏判。未来应引入 LLM-as-Judge 二次校验。
3. **无法验证实际代码正确性**：当前评测仅检查契约结构，不验证生成的代码是否真正
   满足契约语义。这是后续 audit sibling skill 的职责。

## 20. 结论

后续开发这套提示体系时，最重要的不是"写更多规则"，而是持续守住三件事：

1. 先契约，后实现。
2. 不清楚时停下，而不是猜。
3. 只报告真实执行的验证。

只要这三点不丢，这套 skill 就会长期保持稳定价值。
