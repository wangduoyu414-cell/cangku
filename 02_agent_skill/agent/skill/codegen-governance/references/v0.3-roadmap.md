# v0.3 Roadmap Summary

This document summarizes the v0.3 direction for `codegen-governance` under one guiding decision:

> Do not expand the primary language stack. Improve common implementation abilities by deepening scenario coverage.

## Goals

1. Keep the four-pack structure stable: A, B, C, D.
2. Expand high-frequency cross-language scenarios instead of adding new top-level packs.
3. Ensure every new scenario has matching eval assets and at least one example or runnable fixture.
4. Preserve the "contract first, implementation second" and "only report executed validation" principles.

## Delivered Scenario Batches

### Batch 1

- patch / partial update semantics
- outbound HTTP contract
- cache / idempotency / duplicate suppression

### Batch 2

- time / money / precision
- file import / export
- audit redaction / stable diagnostics

### Batch 3

- batch partial success
- async worker / webhook delivery

## Asset Expansion

The current working tree includes:

- `trigger.json`: 20 trigger cases + 16 non-trigger cases
- `scenario-selection-cases.json`: 16 cases
- `tasks.json`: 26 cases
- `examples/`: 8 good-contract examples beyond the original language baseline
- Runnable fixtures in an external or repo-relative fixture library, resolved via `CODEGEN_GOVERNANCE_FIXTURES_ROOT`, legacy `CODEGEN_EXECUTION_CONSTRAINT_FIXTURES_ROOT`, or `--fixtures-root`

## Expected Outcome

By the end of the v0.3 workstream, the skill should be better at common day-to-day implementation risks without becoming a framework encyclopedia:

- clearer PATCH and merge semantics
- better outbound API handling
- stronger cache and idempotency modeling
- safer handling of time, money, files, and audit logs
- explicit batch and worker delivery contracts

## Remaining Work After This Roadmap

The roadmap intentionally does not solve everything. The next logical items remain:

1. `audit` sibling skill for post-generation gatekeeping
2. LLM-as-Judge style quality evaluation for contracts and reports
3. Stronger regression comparison across skill revisions
4. Release/version alignment so version metadata reflects the expanded asset set
