# Pre-Generation Contract Template

Fill this template before producing code.

> **Important**: The Target Language field MUST be filled first. All other fields depend on it.
>
> **Schema note**: Each section header includes an anchor tag (`##-Anchor-Name`) that enables
> programmatic parsing. Do not rename or reorder sections. Anchor tags are stable identifiers
> used by `scripts/eval_contract_output.py` to verify field completeness.
>
> `standard` is the default depth after activation.
> `minimal` is an exception path for whitelist-qualified micro-patches only.
> If any field requires more than a short bullet because of branching, fallback, boundary, or side-effect uncertainty, upgrade to `standard`.
> You may omit SHOULD / MAY sections that are truly irrelevant, but do not omit the MUST anchors.

## Completeness Checklist

Before proceeding to implementation, confirm all MUST fields are filled:

- [ ] **Contract Metadata** - generated_at, task_mode, continuation_mode, and stop_before_code are specified
- [ ] **Contract Depth** - contract_depth is specified as minimal / standard / expanded
- [ ] **Target Language** - language is specified
- [ ] **Scope** - target, change, blast_radius, and non-goals are all present
- [ ] **Selected Scenario Packs** - at least one pack selected with justification
- [ ] **Input Contract** - accepted inputs and invalid/missing handling are specified
- [ ] **Return And Failure Contract** - success, empty, and failure shapes are all specified

If any MUST field is missing or contains only placeholder text like `[todo]` or `[varies]`,
stop and complete the missing information before generating code.

## Minimal Path Checklist

Use the `minimal` depth only when all are true:

- [ ] The change is a micro-edit or single-expression patch.
- [ ] No public contract drift is introduced.
- [ ] No new side-effect scope or trust-boundary behavior is introduced.
- [ ] If the task needs multi-bullet explanation for fallback, boundary, or side-effect ownership, it must be upgraded to `standard`.
- [ ] Any remaining ambiguity can be handled with `continue_with_explicit_assumptions`.

## Continuation Mode Guidance

- Use `continue` when semantics are clear.
- Use `continue_with_explicit_assumptions` only when the assumptions are narrow and low-risk.
- Use `stop_and_ask` when missing information would change behavior, side-effect scope, or public contract.

## Stop Conditions

If any of the following are true after filling this template, do NOT generate code.
Instead, use the stop template and ask for clarification:

1. Required input shape or enum semantics are unknown
2. Return contract would be silently changed by default/fallback
3. Side effects cross file, service, or persistence boundaries without approval
4. Fallback logic would change business meaning
5. Implementation needs a broader design or migration decision

When stopping, output:

```
Cannot proceed safely. Missing: [list the minimum required information].
Stop here. Ask for: [minimum clarification needed].
```
