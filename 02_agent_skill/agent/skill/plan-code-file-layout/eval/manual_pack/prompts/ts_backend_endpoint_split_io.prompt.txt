# Plan Code File Layout

## Overview

Plan file boundaries for one concrete change before implementation starts.
Stay aligned with nearby repository conventions, artifact role, and stack-specific constraints instead of applying generic style advice.

## Quick Reference

- If the target area is unclear, stop and ask for the intended directory, module, or feature anchor.
- If nearby code already shows a stable organization pattern, align with it unless it is clearly causing maintenance or debug pain.
- If the repository shape is unclear, read [repo-shape.md](./references/repo-shape.md) and classify it before planning files.
- If the code role is unclear, choose one artifact path from the `artifact-*` references before reading stack deltas.
- If the repository is mixed, make the decision around the target area and treat other stacks as surrounding constraints rather than primary drivers.

## Workflow

1. Anchor the request to one concrete implementation slice: feature, endpoint, component, command, pipeline step, or module.
2. Read nearby files and infer repository shape with [repo-shape.md](./references/repo-shape.md).
3. Read [core-principles.md](./references/core-principles.md) and pick the closest artifact guide.
4. Read one stack guide that matches the dominant local stack, or treat the area as mixed if multiple stacks materially shape the boundary.
5. Decide `keep`, `split`, or `merge`.
6. Produce a file plan using [output-shape.md](./references/output-shape.md).
7. Explain `why_not_fewer` and `why_not_more`, then list risks or confidence limits.

## Critical Rules

- Default to a small number of high-cohesion files rather than the smallest possible files.
- Split files only when change reasons, side-effect boundaries, test boundaries, or real reuse needs are materially separate.
- Keep private helpers, local types, and one-off mappers inline until they clearly interfere with reading or gain a second real consumer.
- Avoid thin pass-through wrappers whose only job is forwarding calls or re-exporting local logic.
- Keep a common bug or common change readable through roughly one to three files, not a long ping-pong chain.
- Let repository reality win over generic advice unless the existing pattern is obviously harming maintainability or debugging.

## Validation

- Confirm the plan is anchored to one target area rather than drifting into repository redesign.
- Confirm each proposed file has a distinct role and is not a naming-only split.
- Confirm the plan respects nearby directory conventions, package layout, and import style.
- Confirm the plan explains why the boundary is not smaller and not larger.
- Confirm mixed repositories are treated locally rather than forcing one global stack model onto everything.

## Output Contract

Done means:

- The final answer uses the structure in [output-shape.md](./references/output-shape.md).
- The answer returns `keep`, `split`, or `merge`.
- The answer lists the recommended files and each file's responsibility.

## Failure Handling

- If the target area is missing, stop and recover the directory, file, or feature anchor.

- If the task expands into repository-wide restructuring or architecture redesign, hand off to a broader workflow instead of stretching this skill.
- If the stack is unknown, return a repository-aligned generic plan and explicitly mark the stack uncertainty.


# Reference: anti-patterns.md

# Anti-Patterns

Avoid these patterns unless the repository has a very strong reason for them.

## Thin Wrapper Files

Files that only forward calls, re-export local code, or rename a concept without reducing complexity.

## Premature Shared Utils

Creating `shared`, `common`, or `utils` modules before there is a second real consumer.

## Function-Per-File Splits

Breaking one coherent unit into many tiny files so that readers must chase a long import chain.

## Mixed-Layer Giant Files

Keeping route glue, domain logic, persistence, formatting, and recovery logic in one large file.

## Fake Reuse

Extracting code for hypothetical future reuse rather than current duplication or stable API value.

## Cross-File Ping-Pong

A bug or routine change requires repeated jumps across many local files that add little ownership value.

## Package Sprawl

Especially in Go, creating extra packages before there is a real boundary in ownership, dependencies, or API shape.

# Artifact: CLI Command

Use for commands, maintenance tools, and operator-facing scripts.

## Default Boundary

Bias toward fewer files so the main flow is visible in one place.

## Split When

- External integrations or file-system effects deserve isolation.
- Parsing, validation, or transformation logic gains a real test boundary.
- The command shell and the core operation can be separated without hiding the main flow.

## Common Roles

- Command entrypoint
- Core operation
- External adapter

## Avoid

- Spreading a simple command across many modules.
- Abstracting command flags or small helpers into generic frameworks too early.

Use for reusable modules, internal libraries, and shared business logic packages.

## Default Boundary

Keep one cohesive public surface with private helpers nearby until the API or reuse surface grows.

## Split When

- Public API surface and internal helpers are both large enough to obscure each other.
- Multiple stable subdomains emerge inside the module.
- Platform-specific adapters or serialization details deserve isolation.

## Common Roles

- Public entry module
- Core logic module
- Adapter or codec module

## Avoid

- Turning every type or function into its own file.
- Creating a shared library before local duplication or API stability exists.

# Artifact: Service Endpoint

Use for HTTP handlers, RPC handlers, controller actions, and service-facing request flows.

## Default Boundary

Small endpoints may keep handler logic and light orchestration together when persistence and external IO are trivial.

## Split When

- Framework entry glue, business orchestration, and external IO are all present.
- Persistence, network clients, or serialization logic create separate failure modes.
- Contract validation or schema handling has independent test value.

- Handler or controller
- Use-case or service orchestration
- Repository or gateway
- Contract or schema definition

- One-off repositories that only wrap a single local query with no ownership benefit.
- Global service layers that just pass through controller parameters unchanged.

# Artifact: UI Component

Use for pages, feature views, reusable components, and local UI state.

## Default Boundary

Keep view markup, simple local state, and local event handling close together.

## Split When

- Async data access or remote mutations dominate the logic.
- State logic becomes reusable across multiple components.
- One file mixes container concerns with reusable presentational concerns.
- Test strategy clearly separates component rendering from data or side-effect logic.

## Common Roles

- Page or container component
- Reusable presentational component
- Local state or composable hook
- UI-facing service or API adapter

## Avoid

- Splitting every helper into its own file.
- Extracting components only because the template grew, without a real ownership boundary.
- Creating generic shared components before a second real usage appears.


# Reference: core-principles.md

# Core Principles

Use these rules before reading any artifact or stack guide.

## Decision Order

1. Read the target area and nearby files.
2. Classify the local repository shape.
3. Classify the artifact role.
4. Apply stack deltas.
5. Decide `keep`, `split`, or `merge`.

## Default Bias

- Prefer a small number of high-cohesion files.
- Prefer boundaries that make common debugging and common edits local.
- Prefer local helpers over premature extraction.

## Split Signals

Split when one or more of these become materially separate:

- Change reason
- Side-effect boundary
- Test surface
- Reuse need
- Public API surface

Treat stronger signals as:

- External IO or framework glue mixed with business rules
- One file mixing entrypoint, orchestration, and persistence
- One file hiding multiple independent failure modes

## Keep Signals

Keep code in one file when most of these are true:

- The change reason is shared.
- The code is only used locally.
- The same tests cover the whole slice.
- Splitting would add wrapper files or longer jumps without clearer ownership.

## Merge Signals

Merge when the current boundary causes more movement than clarity:

- Frequent bug tracing crosses four or more files.
- A file exists only to forward calls or re-export local code.
- Two files almost always change together and do not have separate test or side-effect value.

## Heuristic Thresholds

Treat these as warning signs, not hard limits:

- A common change should usually stay readable through one to three files.
- Very large files are suspicious when they mix multiple abstraction layers.
- Very small files are suspicious when they exist only for naming or ceremony.

# Reference: repo-shape.md

# Repository Shape

Classify the target area before proposing new files.

## Feature-First

Typical signs:

- Directories grouped by domain or feature
- UI, service, and tests for one feature live near each other

Planning rule:

- Keep new files near the feature.
- Prefer local file splits inside the feature instead of creating global shared folders.

## Layer-First

Typical signs:

- Global folders like `controllers/`, `services/`, `repositories/`, `components/`
- Files for one feature are spread across layer directories

Planning rule:

- Respect the existing layout unless it is clearly being replaced.
- Avoid introducing a one-off feature island inside a strong layer-first codebase.

## Package-First

Typical signs:

- Package boundaries carry most of the meaning
- Common in Go and some Python monorepos

Planning rule:

- First choose the right package.
- Then keep file count modest inside the package unless multiple roles need separation.

## Mixed

Typical signs:

- Different areas follow different patterns
- Frontend, backend, scripts, and infra each organize differently

Planning rule:

- Make the decision locally around the target area.
- Do not force one repository-wide shape onto the entire codebase.
- Let the target artifact type and dominant local stack drive the plan.


# Reference: stack-go.md

# Stack: Go

Treat this as a strong override when Go packages define the main boundary.

## Bias

- Package cohesion matters more than file count.
- A small package with a few files is often healthier than many tiny packages.

## Split When

- The package has clearly separate handler, domain, and external adapter concerns.
- The public API surface is getting hard to scan.
- Import pressure or package cycles point to the wrong package boundary.

## Keep Inline

- Local helpers
- Simple structs used only within one package slice
- Small validation or mapping helpers

## Avoid

- Creating interfaces before there are multiple meaningful consumers
- Splitting packages or files only for abstract purity
- Spreading one flow across many files inside the same package without clear ownership gain


# Reference: stack-python.md

# Stack: Python

Use this guide as a delta on top of the core and artifact rules.

## Bias

- Allow thicker modules than Java-style codebases.
- Prefer package clarity over aggressive file splitting.

## Keep Inline

- Local helpers
- Local data classes or typed dicts
- One-off validators and mappers

## Split When

- Pure logic and external side effects are getting tangled.
- A client, adapter, or persistence edge has clear independent tests.
- The package now exposes a stable API surface used from multiple places.

## Avoid

- One class per file by default
- Catch-all `utils.py` growth
- Over-abstracting small scripts into too many modules


# Reference: stack-typescript-javascript.md

# Stack: TypeScript / JavaScript

Use this guide as a delta on top of the core and artifact rules.

## Bias

- Prefer feature-local organization over global layer sprawl when the repository allows it.
- Keep local types, mappers, and validators close to the feature until reuse is real.

## Split When

- Framework glue, business orchestration, and external IO all coexist in one file.
- The same file carries both UI concerns and remote data concerns.
- A contract, schema, or adapter has its own test and failure boundary.

## Avoid

- Pass-through wrapper files
- Barrel files that add indirection without a public API reason
- Extracting every helper because import paths look cleaner on paper


# Reference: stack-vue.md

# Stack: Vue

Treat this as a strong override when Vue single-file components or composables are involved.

## Bias

- A `.vue` single-file component is already a valid boundary.
- Keep template, simple local state, and local event logic together by default.

## Split When

- Async fetching, mutations, or cross-component state dominate the component.
- State or behavior deserves a reusable composable.
- API or persistence logic is obscuring the view role.

## Common Layout

- Page or container `.vue`
- Reusable child component `.vue`
- Composable for shared stateful behavior
- Service or API module for remote calls

## Avoid

- Turning every piece of component state into a composable
- Creating generic UI abstractions before repeated usage exists
- Splitting one cohesive component just to reduce template length

import { createHmac, timingSafeEqual } from "node:crypto";

type RefundWebhookPayload = {
  refundId: string;
  eventType: "refund.requested" | "refund.completed";
  orderId: string;
};

type HeadersLike = Record<string, string | string[] | undefined>;

type DbLike = {
  execute: (sql: string, params: unknown[]) => Promise<void>;
};

export async function handleRefundWebhook(
  requestBody: string,
  headers: HeadersLike,
  signingSecret: string,
  partnerBaseUrl: string,
  db: DbLike,
): Promise<{ status: number; body: Record<string, string> }> {
  const signature = readSignature(headers);
  if (!signature || !verifySignature(requestBody, signature, signingSecret)) {
    return { status: 401, body: { error: "invalid signature" } };
  }

  const payload = JSON.parse(requestBody) as RefundWebhookPayload;
  const partnerResponse = await fetch(
    `${partnerBaseUrl}/refunds/${encodeURIComponent(payload.refundId)}`,
  );
  if (!partnerResponse.ok) {
    throw new Error(`partner refund lookup failed with status ${partnerResponse.status}`);
  }

  const partnerBody = (await partnerResponse.json()) as {
    refundId: string;
    amountCents: number;
    currency: string;
  };

  await db.execute(
    `
      insert into refund_webhook_events(refund_id, event_type, order_id, amount_cents, currency)
      values (?, ?, ?, ?, ?)
    `,
    [
      payload.refundId,
      payload.eventType,
      payload.orderId,
      partnerBody.amountCents,
      partnerBody.currency,
    ],
  );

  return {
    status: 202,
    body: {
      refundId: payload.refundId,
      eventType: payload.eventType,
    },
  };
}

function readSignature(headers: HeadersLike): string | null {
  const raw = headers["x-partner-signature"];
  if (Array.isArray(raw)) {
    return raw[0] ?? null;
  }
  return typeof raw === "string" ? raw : null;
}

function verifySignature(body: string, signature: string, secret: string): boolean {
  const expected = createHmac("sha256", secret).update(body).digest("hex");
  return timingSafeEqual(Buffer.from(signature), Buffer.from(expected));
}

type RefundPreviewRequest = {
  orderId: string;
  itemCount: number;
  reason: "damaged" | "late" | "customer_remorse";
  expedited: boolean;
};

type RefundPreviewResponse = {
  orderId: string;
  estimatedRefundCents: number;
  estimatedRestockFeeCents: number;
  eligible: boolean;
};

export async function postRefundPreview(requestBody: unknown): Promise<
  | { status: 400; errors: string[] }
  | { status: 200; body: RefundPreviewResponse }
> {
  const parsed = parseRequest(requestBody);
  if (!parsed.ok) {
    return { status: 400, errors: parsed.errors };
  }

  const estimatedRestockFeeCents = estimateRestockFee(
    parsed.value.reason,
    parsed.value.itemCount,
    parsed.value.expedited,
  );
  const estimatedRefundCents = Math.max(2500 - estimatedRestockFeeCents, 0);

  return {
    status: 200,
    body: {
      orderId: parsed.value.orderId,
      estimatedRefundCents,
      estimatedRestockFeeCents,
      eligible: estimatedRefundCents > 0,
    },
  };
}

function parseRequest(
  requestBody: unknown,
): { ok: true; value: RefundPreviewRequest } | { ok: false; errors: string[] } {
  if (!requestBody || typeof requestBody !== "object") {
    return { ok: false, errors: ["body must be an object"] };
  }

  const body = requestBody as Record<string, unknown>;
  const orderId = String(body.orderId ?? "").trim();
  const itemCount = Number(body.itemCount ?? 0);
  const reason = String(body.reason ?? "").trim() as RefundPreviewRequest["reason"];
  const expedited = Boolean(body.expedited ?? false);

  const errors: string[] = [];
  if (!orderId) {
    errors.push("orderId is required");
  }
  if (!Number.isInteger(itemCount) || itemCount <= 0) {
    errors.push("itemCount must be a positive integer");
  }
  if (!["damaged", "late", "customer_remorse"].includes(reason)) {
    errors.push("reason is invalid");
  }

  if (errors.length > 0) {
    return { ok: false, errors };
  }

  return {
    ok: true,
    value: {
      orderId,
      itemCount,
      reason,
      expedited,
    },
  };
}

function estimateRestockFee(
  reason: RefundPreviewRequest["reason"],
  itemCount: number,
  expedited: boolean,
): number {
  if (reason === "damaged") {
    return 0;
  }

  const baseFee = itemCount > 3 ? 900 : 500;
  return expedited ? baseFee + 300 : baseFee;
}

import { sendRefundBatch } from "./refundBatchClient";

type RefundBatchRequest = {
  batchId: string;
  refundIds: string[];
  requestedBy: string;
};

export async function postRefundBatch(
  requestBody: unknown,
  partnerBaseUrl: string,
): Promise<
  | { status: 400; errors: string[] }
  | { status: 202; body: { batchId: string; acceptedCount: number } }
> {
  const parsed = parseRequest(requestBody);
  if (!parsed.ok) {
    return { status: 400, errors: parsed.errors };
  }

  const dispatch = await sendRefundBatch(partnerBaseUrl, parsed.value);
  return {
    status: 202,
    body: {
      batchId: dispatch.batchId,
      acceptedCount: dispatch.acceptedCount,
    },
  };
}

function parseRequest(
  requestBody: unknown,
): { ok: true; value: RefundBatchRequest } | { ok: false; errors: string[] } {
  if (!requestBody || typeof requestBody !== "object") {
    return { ok: false, errors: ["body must be an object"] };
  }

  const body = requestBody as Record<string, unknown>;
  const batchId = String(body.batchId ?? "").trim();
  const requestedBy = String(body.requestedBy ?? "").trim();
  const refundIds = Array.isArray(body.refundIds)
    ? body.refundIds.map((value) => String(value).trim()).filter(Boolean)
    : [];

  const errors: string[] = [];
  if (!batchId) {
    errors.push("batchId is required");
  }
  if (!requestedBy) {
    errors.push("requestedBy is required");
  }
  if (refundIds.length === 0) {
    errors.push("refundIds is required");
  }

  if (errors.length > 0) {
    return { ok: false, errors };
  }

  return {
    ok: true,
    value: {
      batchId,
      refundIds,
      requestedBy,
    },
  };
}

type RefundBatchRequest = {
  batchId: string;
  refundIds: string[];
  requestedBy: string;
};

type RefundBatchDispatch = {
  batchId: string;
  acceptedCount: number;
};

export async function sendRefundBatch(
  partnerBaseUrl: string,
  request: RefundBatchRequest,
): Promise<RefundBatchDispatch> {
  const response = await fetch(`${partnerBaseUrl}/refund-batches`, {
    method: "POST",
    headers: { "content-type": "application/json" },
    body: JSON.stringify(request),
  });

  if (!response.ok) {
    throw new Error(`partner batch dispatch failed with status ${response.status}`);
  }

  return (await response.json()) as RefundBatchDispatch;
}

| Infrastructure → Domain | 基础设施实现领域接口 |
| Shared → 所有层 | 共享工具可被所有层引用 |

**禁止：**
- Domain 层依赖任何外部系统
- 跨越多个层的大文件（如同时处理 HTTP 请求和业务逻辑）
- 违反层间依赖规则的代码

#### 编码规范

- 所有函数和类必须有类型注解
- 遵循各层对应的编码规范（见 `.cursor/rules/`）
- 复杂业务逻辑必须添加注释
- 单个文件不超过 300 行
