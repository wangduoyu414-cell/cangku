# Plan Code File Layout

## Overview

Plan file boundaries for one concrete change before implementation starts.
Stay aligned with nearby repository conventions, artifact role, and stack-specific constraints instead of applying generic style advice.

## Quick Reference

- If the target area is unclear, stop and ask for the intended directory, module, or feature anchor.
- Treat `target_area` as an exact anchor string, not a paraphrase; reuse the concrete path or file anchor from the request instead of rewriting it into a sentence.
- If nearby code already shows a stable organization pattern, align with it unless it is clearly causing maintenance or debug pain.
- If baseline docs explicitly define the target layer, module boundary, or migration direction, follow the baseline docs even when nearby code is off-pattern.
- If baseline docs conflict with each other, prefer the one that most directly defines the target area's current migration state or rollout rule; treat broader architecture summaries as secondary evidence and lower confidence if ambiguity remains.
- If target-area baseline docs agree on the current migration phase but differ on file-boundary granularity, prefer the doc that gives the more concrete file-role guidance for that phase.
- If the repository shape is unclear, read [repo-shape.md](./references/repo-shape.md) and classify it before planning files.
- If the code role is unclear, choose one artifact path from the `artifact-*` references before reading stack deltas.
- If the repository is mixed, make the decision around the target area and treat other stacks as surrounding constraints rather than primary drivers.
- When signals conflict, prefer them in this order: explicit baseline docs for the target area, target file reality, nearby sibling examples, target directory convention, broader repository docs, global architecture docs.
- Within explicit baseline docs, prefer target-area migration notes over broader architecture summaries when they disagree about the current slice.
- Within target-area baseline docs for the same phase, prefer concrete seam guidance over abstract location guidance when choosing whether to keep, split, or merge files.
- Treat file naming as a local clarity choice, not a unique global template; prefer names that express the role clearly inside the target area.
- Treat nearby examples and comparison files as evidence, not as files to include in the plan unless the user explicitly expands the implementation slice to cover them too.

## Workflow

1. Anchor the request to one concrete implementation slice: feature, endpoint, component, command, pipeline step, or module.
2. Read nearby files and infer repository shape with [repo-shape.md](./references/repo-shape.md).
3. Read [core-principles.md](./references/core-principles.md) and pick the closest artifact guide.
4. Read one stack guide that matches the dominant local stack, or treat the area as mixed if multiple stacks materially shape the boundary.
5. Decide `keep`, `split`, or `merge`.
6. Produce a file plan using [output-shape.md](./references/output-shape.md).
7. Explain `why_not_fewer` and `why_not_more`, then list risks or confidence limits.

## Critical Rules

- Default to a small number of high-cohesion files rather than the smallest possible files.
- Split files only when change reasons, side-effect boundaries, test boundaries, or real reuse needs are materially separate.
- Keep private helpers, local types, and one-off mappers inline until they clearly interfere with reading or gain a second real consumer.
- Avoid thin pass-through wrappers whose only job is forwarding calls or re-exporting local logic.
- Keep a common bug or common change readable through roughly one to three files, not a long ping-pong chain.
- Let repository reality win over generic advice only when baseline docs do not explicitly define a different target state.
- If one baseline doc states the long-term target and another gives temporary rollout guidance for the same slice, follow the temporary rollout guidance for the current plan and note the long-term target in `risks`.
- If multiple target-area docs all describe the current phase, follow the one that most concretely assigns file roles, framework glue, or seam ownership for the slice under discussion.
- Hold the core boundary decision stable even when role labels, confidence wording, or local file names could reasonably vary.
- Keep `target_area` exact; do not add explanatory prefixes, suffixes, or migration notes to the anchor field.
- Keep the recommendation scoped to the concrete target slice; do not widen the `files` list to sibling helpers, comparison examples, or the whole directory just because those files were shown as context.
- Never include comparison files, support docs, or nearby evidence files in `files` unless the user explicitly expands the implementation scope to include them.
- If baseline docs and local code conflict, plan toward the documented target state and treat the local layout as legacy evidence to migrate away from.

## Failure Handling

- If the target area is missing, stop and recover the directory, file, or feature anchor.
- Recover missing anchors in this order: explicit path in the request, explicit current file, uniquely named file or directory from the request, one unambiguous nearby candidate from local context. If more than one candidate remains, stop and ask for the anchor.
- If nearby code conflicts with explicit baseline docs, follow the baseline docs and call out relocation or migration cost in `risks`.
- If baseline docs conflict with each other, follow the one that most directly names the target slice's current migration phase, mention the competing long-term target in `risks`, and lower confidence if the cutover state is still uncertain.
- If baseline docs agree on the current phase but disagree on boundary granularity, follow the more concrete seam rule and mention the broader or looser alternative in `risks` when it materially changes the plan.
- If nearby code gives conflicting local signals without an explicit baseline-doc override, follow the strongest signal according to the precedence above and lower confidence.
- If the task expands into repository-wide restructuring or architecture redesign, hand off to a broader workflow instead of stretching this skill.
- If the stack is unknown, return a repository-aligned generic plan and explicitly mark the stack uncertainty.


# Reference: core-principles.md

# Core Principles

Use these rules before reading any artifact or stack guide.

## Decision Order

1. Read the target area and nearby files.
2. Classify the local repository shape.
3. Classify the artifact role.
4. Apply stack deltas.
5. Decide `keep`, `split`, or `merge`.

## Conflict Precedence

When local signals disagree, prefer them in this order:

1. Explicit baseline docs or migration rules that define the target area
2. The target file and the concrete implementation slice itself
3. Nearby sibling examples in the same local area
4. The target directory's existing convention
5. Broader repository documentation
6. Global architecture guidance

If explicit baseline docs conflict with local structure, follow the documented target state and treat the local layout as legacy evidence. If the winning signal is still ambiguous, lower confidence and call out migration cost or uncertainty explicitly.

When explicit baseline docs conflict with each other, prefer the one that most directly defines the target slice's current migration phase or rollout status. Use broader architecture summaries as secondary evidence, and call out the unresolved long-term target in `risks` when the docs describe different phases of the same migration.

When target-area baseline docs agree on the current phase but differ on file-boundary granularity, prefer the one that gives the more concrete seam guidance for the slice itself. A doc that names which file should own framework glue, response shaping, or adapter behavior should beat a doc that only names the broader destination directory.

## Default Bias

- Prefer a small number of high-cohesion files.
- Prefer boundaries that make common debugging and common edits local.
- Prefer local helpers over premature extraction.
- Prefer the shortest file chain that still exposes real side-effect boundaries.
- Do not insert an extra orchestration layer when the entry file plus the real IO edges already keeps the common path readable.
- Keep the recommendation scoped to the concrete slice under discussion; nearby examples are there to calibrate the decision, not to be copied into the returned file plan.
- When baseline docs explicitly define a different layer or module boundary, prefer the documented target state over preserving an off-pattern local layout.

## Split Signals

Split when one or more of these become materially separate:

- Change reason
- Side-effect boundary
- Test surface
- Reuse need
- Public API surface

Treat stronger signals as:

- External IO or framework glue mixed with business rules
- One file mixing entrypoint, orchestration, and persistence
- One file hiding multiple independent failure modes
- One file where the real split is "entry + external adapter + persistence adapter", not "entry + service + adapter"

## Keep Signals

Keep code in one file when most of these are true:

- The change reason is shared.
- The code is only used locally.
- The same tests cover the whole slice.
- Splitting would add wrapper files or longer jumps without clearer ownership.
- The main flow is still readable in the entry file once real adapter boundaries are extracted.

## Merge Signals

Merge when the current boundary causes more movement than clarity:

- Frequent bug tracing crosses four or more files.
- A file exists only to forward calls or re-export local code.
- Two files almost always change together and do not have separate test or side-effect value.

## Heuristic Thresholds

Treat these as warning signs, not hard limits:

- A common change should usually stay readable through one to three files.
- Very large files are suspicious when they mix multiple abstraction layers.
- Very small files are suspicious when they exist only for naming or ceremony.


# Reference: repo-shape.md

# Repository Shape

Classify the target area before proposing new files.

## Feature-First

Typical signs:

- Directories grouped by domain or feature
- UI, service, and tests for one feature live near each other

Planning rule:

- Keep new files near the feature.
- Prefer local file splits inside the feature instead of creating global shared folders.

## Layer-First

Typical signs:

- Global folders like `controllers/`, `services/`, `repositories/`, `components/`
- Files for one feature are spread across layer directories

Planning rule:

- Respect the existing layout unless it is clearly being replaced.
- Avoid introducing a one-off feature island inside a strong layer-first codebase.

## Package-First

Typical signs:

- Package boundaries carry most of the meaning
- Common in Go and some Python monorepos

Planning rule:

- First choose the right package.
- Then keep file count modest inside the package unless multiple roles need separation.

## Mixed

Typical signs:

- Different areas follow different patterns
- Frontend, backend, scripts, and infra each organize differently

Planning rule:

- Make the decision locally around the target area.
- Do not force one repository-wide shape onto the entire codebase.
- Let the target artifact type and dominant local stack drive the plan.

      role: ""
      responsibility: ""

  avoid: []
  why_not_fewer: ""
  why_not_more: ""
  risks: []
  confidence: high | medium | low
```

## Field Notes

### Core Decision Fields

- `strategy`: Main boundary decision for this implementation slice.
- `target_area`: Directory, feature, file, or module anchor used for the decision.
- `target_area` should copy the concrete anchor from the request exactly rather than turning it into a prose label or migration summary.
- `files`: Recommended production files only.
- `files` should list only the production files for the target slice itself, not sibling examples or comparison files that were shown as context.
- `files` should also exclude support docs, baseline docs, and repository context files that were injected only to guide the decision.
- When baseline docs conflict with the local layout, `files` should follow the documented target state and `risks` should note migration or relocation impact.
- `why_not_fewer`: Explain why a smaller layout would blur a real boundary.
- `why_not_more`: Explain why further splitting would mostly add ceremony.

These fields carry the main implementation decision and should remain stable across reasonable model variations.

### Explanatory Fields

- `repo_shape`: Local repository organization around the target area.
- `artifact_type`: Primary code role that drives the boundary.
- `stack`: Dominant local stack, not the entire repository.
- `keep_inline`: Helpers, types, validators, or mappers that should stay local for now.
- `avoid`: Unhelpful files or abstractions that should not be created.
- `risks`: List uncertainty, convention conflicts, or likely follow-up questions.
- `confidence`: Record how stable the recommendation looks after weighing local signals and conflicts.

These fields explain or classify the decision. They should still be present, but some wording, naming, or classification differences can remain acceptable when the core boundary decision is unchanged.


# Reference: artifact-service-endpoint.md

# Artifact: Service Endpoint

Use for HTTP handlers, RPC handlers, controller actions, and service-facing request flows.

## Default Boundary

Small endpoints may keep handler logic and light orchestration together when persistence and external IO are trivial.

## Split When

- Framework entry glue, business orchestration, and external IO are all present.
- Persistence, network clients, or serialization logic create separate failure modes.
- Contract validation or schema handling has independent test value.
- The route can stay as the visible orchestration file while external adapters take the real IO boundaries.

## Common Roles

- Handler or controller
- Use-case or service orchestration
- Repository or gateway
- Contract or schema definition

## Avoid

- One-off repositories that only wrap a single local query with no ownership benefit.
- Global service layers that just pass through controller parameters unchanged.
- Feature-local service files whose only extra value is moving a short route orchestration out of the handler while client and store files already hold the real boundaries.
- Route -> service -> client -> repository chains when the common endpoint path is still readable through route plus the real adapters.


# Reference: stack-python.md

# Stack: Python

Use this guide as a delta on top of the core and artifact rules.

## Bias

- Allow thicker modules than Java-style codebases.
- Prefer package clarity over aggressive file splitting.

## Keep Inline

- Local helpers
- Local data classes or typed dicts
- One-off validators and mappers

## Split When

- Pure logic and external side effects are getting tangled.
- A client, adapter, or persistence edge has clear independent tests.
- The package now exposes a stable API surface used from multiple places.

## Avoid

- One class per file by default
- Catch-all `utils.py` growth
- Over-abstracting small scripts into too many modules

## 变更管理

### 小改动

- 修改单个文件或函数
- 不影响模块边界
- 不需要更新文档

处理流程：
1. 若层归属或职责边界不清，补充语义上下文（describe-module-context）
2. 编码
3. 验证

### 中等改动

- 涉及多个文件
- 可能影响模块边界
- 需要更新相关文档

处理流程：
1. 若层归属或职责边界不清，补充语义上下文（describe-module-context）
2. 若依赖关系或影响面不清，分析依赖（analyze-current-file-dependencies-workflow）
3. 编码
4. 更新 `docs/decisions/CHANGELOG.md`
5. 验证

### 大改动（架构变更）

- 涉及模块重构
- 改变层间依赖关系

```text
# Clean Architecture Layering Rules

本文档定义本仓库的 Clean Architecture 层次结构、各层职责边界、层间依赖规则和编码规范。

## 目录结构

```
src/app/
├── entrypoints/       # 入口点层：外部系统与应用的交互入口
├── interfaces/        # 接口层：适配器、控制器、网关
├── application/       # 应用层：用例编排、DTO、接口定义
├── domain/           # 领域层：实体、值对象、领域服务、领域事件
├── infrastructure/   # 基础设施层：数据库、外部服务、消息队列
├── shared/           # 共享层：工具函数、常量、通用组件
└── config/          # 配置层：配置加载、环境变量
```

## 层概览

| 层名称 | 目录 | 核心职责 | 依赖方向 |
|--------|------|---------|---------|
| EntryPoints | `entrypoints/` | 外部系统交互入口 | → Interfaces, Application, Domain |
| Interfaces | `interfaces/` | 适配器与网关 | → Application, Domain |
| Application | `application/` | 用例编排 | → Domain, Infrastructure |
| Domain | `domain/` | 核心业务规则 | 无（最内层） |
| Infrastructure | `infrastructure/` | 技术实现 | → Domain |
| Shared | `shared/` | 跨层复用 | 允许所有层引用 |
| Config | `config/` | 配置管理 | 仅加载，不依赖业务层 |

---

## 1. EntryPoints 层（入口点层）

### 1.1 核心职责

- 作为外部系统与应用的交互入口
- 处理协议转换（HTTP、CLI、消息队列、WebSocket 等）
- 接收用户输入，调用下层接口，返回响应
- **不包含任何业务逻辑**

### 1.2 允许的文件类型

```
entrypoints/
├── cli/              # 命令行入口
│   └── commands/     # CLI 命令实现
├── http/             # HTTP 入口
│   └── routes/      # 路由定义
├── graphql/          # GraphQL 入口
├── websocket/        # WebSocket 入口
└── consumer/         # 消息消费者
```

### 1.3 编码规范

**必须遵守：**
- 不包含业务逻辑
- 不直接操作数据库
- 不直接调用外部服务
- 使用依赖注入获取业务层服务
- 所有请求处理函数必须包含类型注解
- 错误处理返回有意义的 HTTP 状态码或错误码

**建议遵守：**
- 每个入口点文件不超过 100 行
- 使用统一的错误响应格式
- 请求参数使用 Pydantic 或等效库验证

**文件命名：**
- CLI 命令：`{action}_command.py`
- HTTP 路由：`{resource}_routes.py`
- 消费者：`{event}_consumer.py`

### 1.4 层间依赖

```
EntryPoints
    ├──→ Interfaces    # 可选：使用接口层适配器
    ├──→ Application  # 主要：调用 UseCase
    └──→ Domain       # 禁止直接调用 Domain 实体
```

---

## 2. Interfaces 层（接口层）

### 2.1 核心职责

- 定义应用与外部世界交互的接口
- 实现端口（Port）定义，供 Infrastructure 层实现
- 协议相关的类型定义
- **不包含业务逻辑，但可以包含接口验证逻辑**

### 2.2 允许的文件类型

```
interfaces/
├── controllers/      # 控制器：处理特定业务领域的请求
├── gateways/        # 网关：定义与外部系统交互的端口
├── dto/             # 数据传输对象：请求/响应结构
├── adapters/        # 适配器：实现网关接口
└── presenters/      # 展示器：格式化输出
```

### 2.3 编码规范

**必须遵守：**
- 网关接口（Port）在 interfaces 层定义
- 适配器实现（Adapter）在 Infrastructure 层实现
- DTO 使用 Pydantic 或等效库定义
- 不包含业务逻辑判断

**建议遵守：**
- 每个控制器对应一个 Application 层的 UseCase
- DTO 命名使用 `{Entity}{Action}Request` / `{Entity}{Action}Response` 格式
- 网关接口命名使用 `I{ExternalSystem}Gateway` 格式

**文件命名：**
- 控制器：`{entity}_controller.py`
- 网关：`i_{system}_gateway.py`
- DTO：`{entity}_{action}_dto.py`
- 适配器：`{system}_gateway_adapter.py`

### 2.4 层间依赖

```
Interfaces
    ├──→ Application  # 主要：调用 UseCase
    └──→ Domain       # 可选：引用实体类型
```

---

## 3. Application 层（应用层）

### 3.1 核心职责

- 用例（UseCase）编排
- 协调多个 Domain 对象完成业务操作
- 定义应用服务的接口
- 管理事务边界
- **包含应用逻辑（流程编排），不包含领域业务规则**

### 3.2 允许的文件类型

```
application/
├── use_cases/        # 用例实现
│   └── {entity}_use_cases.py
├── services/         # 应用服务接口
│   └── {entity}_service.py
├── ports/            # 端口定义（被 Infrastructure 实现）
│   └── {repository}_port.py
└── dto/              # 应用层 DTO
```

### 3.3 编码规范

**必须遵守：**
- 每个用例对应一个用户操作（如创建订单、查询用户）
- 使用依赖注入获取 Domain 服务和 Infrastructure 端口
- 方法签名包含输入输出类型注解
- 用例失败时抛出应用层异常（不使用领域层异常）

**建议遵守：**
- UseCase 类以 `UseCase` 结尾
- UseCase 方法统一使用 `execute(input: InputDTO) -> OutputDTO` 签名
- 每个 UseCase 不超过 200 行
- 使用 Pydantic 定义输入输出 DTO
- 用例应该有对应的单元测试

**文件命名：**
- 用例：`create_{entity}_use_case.py`
- 应用服务：`{entity}_service.py`
- 端口：`i_{entity}_repository.py`

### 3.4 层间依赖

```
Application
    ├──→ Domain       # 主要：调用领域服务和实体
    └──→ Infrastructure # 主要：注入并使用仓储端口
```

## 4. Domain 层（领域层）

### 4.1 核心职责

- 封装核心业务规则和实体
- 定义业务 invariants（不变式）
- 包含领域事件
- **不依赖任何外部系统，是最内层**

### 4.2 允许的文件类型

```
domain/
├── entities/         # 实体
│   └── {entity}.py
├── value_objects/    # 值对象
│   └── {value_object}.py
├── services/         # 领域服务
│   └── {domain}_service.py
├── events/           # 领域事件
│   └── {event}.py
├── repositories/     # 仓储接口（仅接口定义）
│   └── i_{entity}_repository.py
└── exceptions/       # 领域异常
    └── {domain}_exception.py
```

### 4.3 编码规范

**必须遵守：**
- 不直接引入数据库操作
- 不处理 HTTP 请求/响应
- 不包含日志记录逻辑
- 不依赖 Infrastructure 层
- 使用类型注解标注所有方法签名
- 实体必须有唯一标识（ID）
- 状态变更通过显式方法，不直接修改属性

**建议遵守：**
- 每个实体不超过 300 行
- 每个方法不超过 30 行
- 实体属性使用 property 暴露（可选读/写控制）
- 使用领域事件记录重要状态变更
- 复杂业务规则使用领域服务而非在实体中堆砌

**文件命名：**
- 实体：`{entity}.py`
- 值对象：`{value_object}.py`
- 领域服务：`{domain}_service.py`
- 领域事件：`{entity}_{event}_event.py`
- 仓储接口：`i_{entity}_repository.py`

### 4.4 层间依赖

```
Domain
    └── 无（最内层，不依赖任何层）
```

### 4.5 常见模式

**实体设计：**
```python
class Order:
    def __init__(self, order_id: str, customer_id: str):
        self._id = order_id
        self._customer_id = customer_id
        self._status = OrderStatus.PENDING
        self._items: list[OrderItem] = []
        self._validate_invariants()

    @property
    def id(self) -> str:
        return self._id

    def place(self) -> None:
        if self._status != OrderStatus.PENDING:
            raise InvalidOrderStateError("Only pending orders can be placed")
        self._status = OrderStatus.PLACED
        self.add_event(OrderPlacedEvent(order_id=self._id))

    def add_item(self, item: OrderItem) -> None:
        self._items.append(item)
        self._recalculate_total()

    def _validate_invariants(self) -> None:
        if not self._items:
            raise InvalidOrderError("Order must have at least one item")
```

## 层归属判定规则

| 文件路径模式 | 架构层 | 说明 |
|------------|--------|------|
| `src/app/entrypoints/` | EntryPoints | 入口点层 |
| `src/app/interfaces/` | Interfaces | 接口层 |
| `src/app/application/` | Application | 应用层 |
| `src/app/domain/` | Domain | 领域层 |
| `src/app/infrastructure/` | Infrastructure | 基础设施层 |
| `src/app/shared/` | Shared | 共享层 |
| `src/app/config/` | Config | 配置层 |
| `tests/unit/` | Unit Tests | 单元测试（针对某层） |

## refunds

### Basic Info
- **module path**: `src/app/{domain,application,interfaces,infrastructure}/refunds/`
- **architecture layers**: Domain, Application, Interfaces, Infrastructure
- **status**: active
- **owner**: operations-platform

### Core Responsibility
Own the refund review flow, including request validation, approval policy, orchestration, API exposure, and persistence adapters.

### Boundary
- **In scope**: refund request modeling, approval policy checks, use case orchestration, HTTP input/output translation, repository adapters.
- **Out of scope**: payment capture, ledger posting, and partner settlement export.
- **Layer split**:
  - Domain keeps refund rules and invariants.
  - Application coordinates approval flow.
  - Interfaces exposes controller or presenter shapes.
  - Infrastructure stores or fetches refund records.

### Public Interfaces
| interface | purpose | called by |
|-----------|---------|-----------|
| `RefundPolicyService` | decide whether a request can be auto-approved | Application |
| `create_refund` use case | create and normalize a refund request | Interfaces / EntryPoints |
| `RefundRepository` | persist refund snapshots | Application |

### Dependencies
| dependency | reason | type |

| `config.refund_settings` | load operational limits | optional |

### Downstream Consumers
| consumer | usage |
|----------|-------|

| `interfaces/refunds/refund_controller.py` | adapt incoming requests to the use case |
| `scripts/ops/reconcile_refund_backlog.py` | operational follow-up on pending refunds |

### Migration Notes
- `src/app/entrypoints/http/refund_routes.py` is the target home for new refund HTTP transport work.

- `src/app/infrastructure/refunds/refund_repository.py` is a pre-migration path kept only for compatibility and should not be the target for new production logic.
