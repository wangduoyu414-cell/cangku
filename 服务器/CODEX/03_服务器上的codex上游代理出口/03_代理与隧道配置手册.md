# 代理与隧道配置手册

本文档记录当前系统真实上游调用的关键配置：独立出口机、业务服务器本机 `sing-box` 客户端、容器代理配置与验证方法。

这是当前系统能否真正调用上游的核心文档。

## 1. 当前架构结论

当前业务服务器 **不能直连** `chatgpt.com`。

真实可用链路依赖下面这条路径：

```text
业务服务器容器
  -> http://10.2.0.15:17891
  -> 业务服务器本机 sing-box 客户端
  -> 远端出口机 216.167.22.199:49490
  -> 外部网络
  -> chatgpt.com
```

只要这条链路断开，业务服务器上的真实 API 调用就会失效。

文件：

```text
/etc/systemd/system/sing-box-egress.service
```

当前实际内容：

```ini
[Unit]
Description=sing-box Egress Client
After=network.target nss-lookup.target

[Service]
User=root
NoNewPrivileges=true
ExecStart=/usr/bin/sing-box run -c /etc/sing-box-egress/config.json
Restart=on-failure
RestartPreventExitStatus=23
LimitNPROC=10000
LimitNOFILE=1048576
PrivateTmp=true

[Install]
WantedBy=multi-user.target
```

## 6. 故障模式与判断方法

### 6.1 业务服务器本机客户端未运行

表现：

- 服务器 `curl -x http://10.2.0.15:17891 ...` 连接失败
- 服务日志中出现 `error sending request for url`
- 最终请求收口为 `504 upstream total timeout exceeded`

### 6.2 出口机不可达或节点参数错误

表现：

- `sing-box-egress` 启动失败
- 或 `curl -x http://10.2.0.15:17891 ...` 直接失败
- 或 `sing-box-egress` 日志里出现握手失败

### 6.3 代理通了但上游挑战

表现：

- `curl -x` 返回 `403` / `Cf-Mitigated: challenge`
- 这说明网络链路通，但业务认证或指纹策略被 Cloudflare 挑战
- 对于裸探测是正常的，不能直接据此判断代理无效

### 6.4 业务服务器直连不可用

当前已确认业务服务器直连失败：

```text
curl https://chatgpt.com   -> timeout
curl -4 https://chatgpt.com -> timeout
curl -6 https://chatgpt.com -> connect failed
```

所以当前系统不应依赖业务服务器自带出口。

## 7. 如何切换到新的代理地址

### 7.1 如果继续沿用独立出口机方案

需要改两处：

1. 业务服务器 `/etc/sing-box-egress/config.json`
2. 业务服务器 `docker-compose.server.yml` 中的 `CODEXMANAGER_UPSTREAM_PROXY_URL`

### 7.2 如果以后换另一台出口机

则可以把：

```text
CODEXMANAGER_UPSTREAM_PROXY_URL
```

改成：

```text
http://10.2.0.15:<NEW_LOCAL_PORT>
```

同时修改 `sing-box-egress` 的远端节点参数。

## 8. 安全注意事项

### 8.1 当前 17891 监听在 `10.2.0.15`

这是为了让容器可通过宿主机内网地址访问本机客户端。

当前之所以可接受，是因为：

- 腾讯云防火墙未开放 `17891`
- 它只监听内网地址，不监听公网地址

后续不要误把 `17891` 加入云防火墙。

### 8.2 文档不记录敏感值

本文档不记录：

- SSH 私钥内容
- 实际平台 `API Key`
- 账号 token 明文
